# ADR 0004 — Completions API Design: Proto Schema, Tensor Encoding, and Embedding Corpus

**Date**: 2026-05-03
**Status**: Accepted
**Branch**: `010-phase-6`

---

## Context

Phase 6 adds a `/v1/completions` endpoint that accepts both plain-text prompts and pre-computed
prompt embedding tensors. Three design questions required explicit decisions before implementation:

1. How should the proto schema handle the two mutually-exclusive input types?
2. How should prompt embedding tensors be encoded on the wire?
3. How should the benchmark embedding corpus be structured?

---

## Decision 1: Proto Schema — `oneof input` with `bytes` for `prompt_embeds`

### Choice

```proto
message CompletionRequest {
  string model      = 1;
  int32  max_tokens = 2;
  optional float temperature = 3;
  optional float top_p       = 4;
  optional int64 seed        = 5;
  oneof input {
    string prompt        = 6;
    bytes  prompt_embeds = 7;
  }
}
```

The proto3 `oneof input` block enforces mutual exclusivity of `prompt` and `prompt_embeds` at
the schema level. The `bytes` field carries raw `torch.save()` output with no additional encoding.

### Rationale

- **`oneof` enforcement**: Setting both fields is impossible in the generated API; the runtime
  silently drops the earlier-set field. Application code still checks `WhichOneof("input")` to
  distinguish text vs. tensor paths and returns `INVALID_ARGUMENT` if neither is set.
- **`bytes` over `string`**: Arbitrary binary data is correctly represented as `bytes`. Using
  `string` would be a protobuf anti-pattern (proto3 `string` must be valid UTF-8; tensor bytes
  are not).
- **No base64 inside proto**: The protobuf wire format natively encodes `bytes` fields without
  base64 overhead. This is the principal wire-efficiency advantage over the REST path, where
  JSON forces base64 encoding (~33% overhead across all sequence lengths).

### Alternatives Considered

- **Two separate RPCs** (`CompleteTxt` / `CompleteEmbed`): Rejected — doubles the service surface
  and prevents a single client-side call to handle both inputs uniformly.
- **`string prompt_embeds` (base64)**: Rejected — both wrong type and negates the efficiency
  advantage this phase is designed to measure.

## Decision 2: Tensor Encoding Convention — `torch.save()` / `torch.load()`

### Choice

Tensors are serialised with `torch.save(tensor, buf)` into a `BytesIO` buffer and
deserialised with `torch.load(io.BytesIO(raw_bytes), weights_only=True)`.

- **Proxy → proto**: The proxy receives a base64-encoded `torch.save()` string from the
  OpenAI-compatible REST client, base64-decodes it to raw bytes, and writes those bytes into the
  proto `CompletionRequest.prompt_embeds` field.
- **Frontend**: The frontend reads the raw bytes from the proto field, calls
  `torch.load(io.BytesIO(raw_bytes), weights_only=True)`, validates `dtype` ∈
  {float32, bfloat16, float16} and `ndim == 2`, then passes the reconstructed tensor to
  `engine.generate({"prompt_embeds": tensor}, ...)`.
- **Client library**: `VllmGrpcClient.completions.complete(prompt_embeds=tensor)` serialises
  the tensor via `torch.save()` into the proto bytes field directly — no base64 anywhere in
  the gRPC path.

### Rationale

`torch.save()` is vLLM's own persistence format for checkpoints and is the natural
serialisation round-trip for `torch.Tensor`. The `weights_only=True` flag avoids arbitrary
code execution from maliciously crafted serialised objects (security hardening).

### Alternatives Considered

- **numpy `frombuffer` / raw float bytes**: Requires agreeing on dtype, endianness, and shape
  out-of-band. `torch.save()` encodes all metadata inside the payload.
- **ONNX or safetensors**: Adds a dependency; no benefit for single-tensor transport.

## Decision 3: Embedding Corpus Design — 4-Bucket, 20-Sample Approach

### Choice

The benchmark corpus contains 20 pre-computed `.pt` tensor files across four sequence-length
buckets drawn from the existing `chat_nonstreaming.json` corpus:

| Bucket | seq_len range | Samples | Source |
|--------|--------------|---------|--------|
| short  | 8–16         | 5       | Direct extraction from single chat entries |
| medium | 32–48        | 5       | Direct extraction from single chat entries |
| long   | 96–128       | 5       | Adjacent chat entries concatenated |
| full   | 192–256      | 5       | Adjacent chat entries concatenated |

Each `.pt` file is generated by `scripts/python/gen_embed_corpus.py` using the Qwen3-0.6B
tokenizer and `model.embed_tokens` CPU layer. The manifest (`manifest.json`) records
`source_prompt`, `seq_len`, `shape`, `dtype`, `bucket`, and `source` for every entry.
Prompts are also written to `prompts.txt` for independent re-generation.

### Rationale

- **4 buckets**: Demonstrates that the ~33% base64 overhead is consistent across payload sizes,
  not an artefact of a single benchmark point. Each bucket spans a realistic token range.
- **20 samples**: Enough for stable mean wire-size measurements without excessive generation time.
  The integer 20 divides evenly across 4 buckets (5 each), keeping the report balanced.
- **Sourced from existing corpus**: Re-using `chat_nonstreaming.json` avoids introducing a
  second corpus maintenance burden. Long/full buckets are built by concatenating adjacent entries,
  and the `source` field in the manifest records which entries were joined.
- **CPU generation**: `model.embed_tokens` (the embedding lookup layer) runs on CPU without a GPU,
  making corpus generation reproducible in any development environment.

### Alternatives Considered

- **Random tensors**: Would not reflect realistic model-input distributions; wire-size
  measurements would be identical to random noise (no structured sparsity).
- **Single bucket (fixed seq_len)**: Would leave open the question of whether the efficiency
  advantage holds at other sequence lengths.
- **Larger corpus (100+ samples)**: Increases commit size and generation time with diminishing
  informational benefit for the wire-size comparison.
