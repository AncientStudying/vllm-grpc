# Feature Specification: M6.1.1 — Engine-Cost Instrumentation Diagnosis & Symmetrisation

**Feature Branch**: `023-m6-1-1-engine-cost-instrumentation`
**Created**: 2026-05-16
**Status**: Draft
**Input**: User description: "M6.1.1 as described in PLAN.md"

## Background *(non-normative summary)*

M6.1's full-sweep verdict report ([docs/benchmarks/m6_1-real-prompt-embeds.md](../../docs/benchmarks/m6_1-real-prompt-embeds.md)) raised `engine_cost_drift_warning` on **all three chat_stream cells** (`c=1`, `c=4`, `c=8` at `hidden_size=4096`). The per-cohort spread on `engine_ttft_ms` was consistent across all three cells at roughly 14–17%: `rest_https_edge ≈ 43.5 ms / default_grpc ≈ 47.5 ms / tuned_grpc_multiplexed ≈ 41.5 ms`. The same engine code path served every RPC in those cells; only the transport differed. So either:

- **Hypothesis A — Measurement-window asymmetry.** REST captures `engine_start = perf_counter()` inside the FastAPI shim (a→ first SSE chunk yielded), while gRPC reads `engine_ttft_ms` from server-side trailing metadata anchored on a slightly different point. The two clocks straddle different windows; the engine is doing the same work but the reported number is bracketing different segments. This would be a measurement artifact: harmless to the engine but contaminating any per-cohort engine-cost decomposition published from M6.1 onward.
- **Hypothesis B — Channel-dependent batching.** vLLM's continuous batching sees REST and gRPC arrival patterns differently (HTTPS edge buffers requests with different jitter than a raw TCP gRPC channel), and the engine schedules them into different batches with measurably different first-token latencies. This would be a real effect, not an artifact — operationally significant: it means protocol choice changes the engine's behavior, not just the wire-cost differential.

M6.1.1 distinguishes the two with a multi-point timing diagnosis sweep, then either fixes the instrumentation (Hypothesis A) or publishes the finding as a methodology supersedence (Hypothesis B). The decision is data-driven, not pre-committed.

**Why M6.1.1 must close before M6.2 (token-budget axis) opens.** M6.2 introduces a `max_tokens` measurement axis. Bundling instrumentation work with the axis change would make the M6.2 diff against M6.1 ambiguous (did the per-cohort engine_ttft shift because of longer generation, or because the measurement window was fixed?) — exactly the methodological confound Spec Kit Phase Discipline (one-variable-per-milestone) is designed to prevent. Two further concrete reasons: (1) M6.2's `max_tokens=10` / `50` null anchors must reproduce M6.1's measurements within published CIs to validate sweep integrity; if the instrumentation gap is constant, every anchor on every cap will fire `engine_cost_drift_warning` regardless of whether the rest of the sweep is sound, and the anchor mechanism stops working. (2) The same gap will recur in M7 (corpus diversity) and M8 (multi-model); every per-cohort engine-cost decomposition published from now on will carry a 14–17% asterisk until it is resolved. Cheap to fix once now; expensive to retrofit later.

## Clarifications

### Session 2026-05-16

- Q: How does FR-010 measure "≥80% of the M6.1-reported cell-level cohort spread maps onto `seg_ab`" (or `seg_bc`)? → A: Magnitude-equivalence classifier (option B). For each chat_stream cell, compute `spread(x) = max(x) − min(x)` over the three cohort means. The cell classifies as `instrumentation_artifact` iff `spread(seg_ab) / spread(engine_ttft) ≥ 0.80`; as `channel_dependent_batching` iff `spread(seg_bc) / spread(engine_ttft) ≥ 0.80`; as `drift_not_reproduced` iff `spread(engine_ttft) / mean(engine_ttft) < 0.05` (the cell-level Phase 1 spread is below 5%, i.e. M6.1's reported drift is not reproduced at n=50); else `inconclusive`. Operator can verify the classification by reading the multi-point timing table and applying the formula by hand — no rank-correlation or regression machinery required. Mirrors the same `max − min` shape M6.1 already used to flag `engine_cost_drift_warning` (FR-022 in M6.1).
- Q: If Phase 2(a) verification fires `chat_stream_control_drift_warning` because the symmetrised instrumentation shifted `engine_ttft_ms` enough that the new n=100 CIs no longer overlap M6.1's published CIs, what is the operator action? → A: Phase 2(a) publishes a fresh chat_stream baseline (option D). The verification sweep MUST emit a new top-level JSON section `chat_stream_baseline_post_symmetrisation` containing the per-(cell × cohort) `engine_ttft_ms` and `engine_tpot_ms` means + 95% CIs measured under the symmetrised instrumentation. M6.2 (and any downstream milestone using chat_stream as a null anchor) anchors against THIS section, not M6.1's pre-symmetrisation numbers. M6.1's chat_stream verdicts remain valid AS PUBLISHED — they were correct under M6.1's instrumentation; the supersedence is methodology-only (the brackets changed), not verdict-revocation. The `chat_stream_control_drift_warning` flag is recorded in M6.1.1's report with an explanatory annotation: "expected — reflects bracketing change in Phase 2(a) symmetrisation; not infrastructure drift". The fresh baseline section is the operator-facing artifact that closes the gap M6.1.1 was created to address.
- Q: If Phase 1's three chat_stream cells return mixed classifications (e.g., two `instrumentation_artifact` and one `channel_dependent_batching`), what does the operator do? → A: Re-run Phase 1 once before committing to any Phase 2 path (option B). The mini-sweep is cheap (~30 min, <$1) and "diagnose first" is the discipline that motivated M6.1.1 to exist. The harness MUST detect mixed classifications across the three chat_stream cells and refuse to advance to Phase 2 — exit code `3` with a stderr message naming the divergent cells and instructing the operator to re-run `--m6_1_1-diagnose`. After the second Phase 1 run: (a) if classifications agree across all three cells, proceed with the agreed Phase 2 path; (b) if classifications still diverge after re-confirmation, the operator chooses between per-cell action (apply the fix to `instrumentation_artifact` cells and document `channel_dependent_batching` cells in a heterogeneous Phase 2 outcome) or splitting M6.1.1 into successor sub-milestones — the choice is recorded in `phase_2_choice` with a brief (≤2 sentence) justification. `drift_not_reproduced` cells do NOT count as divergent for this check (they short-circuit the segment-level classification per FR-010); a cell-set of {`drift_not_reproduced`, `instrumentation_artifact`, `instrumentation_artifact`} is treated as uniform `instrumentation_artifact` for the Phase 2 path decision.
- Q: If Phase 1 returns `drift_not_reproduced` on all three chat_stream cells (M6.1's reported 14–17% spread is not reproduced at n=50), what action does the operator take on M6.1's published `engine_cost_drift_warning` flag? → A: Re-confirm with a second Phase 1 run before declaring `drift_not_reproduced` final (option C). The harness MUST refuse to close M6.1.1 on a single `drift_not_reproduced` Phase 1 run with the same exit-code-`3` "re-run `--m6_1_1-diagnose`" message. After the second `--m6_1_1-diagnose` run: (a) if the second run also returns uniform `drift_not_reproduced`, M6.1's `engine_cost_drift_warning` flag is preserved AS PUBLISHED (it correctly reflected M6.1's measurement at the time), AND M6.1's JSON gains a `methodology_supersedence` note recording that "the drift warning was not reproduced in two independent n=50 Phase 1 mini-sweeps on $(date)" — downstream readers can weight the M6.1 flag accordingly. The M6.1.1 report's Phase 2 path is recorded as `drift_not_reproduced_confirmed`; no code change is applied and no doc-only `contracts/instrumentation.md` update is written (the flag's interpretation is preserved as-is — neither symmetrised nor documented as a real effect). (b) If the second run returns a segment-level classification (`instrumentation_artifact` or `channel_dependent_batching`), the first run is treated as transient and the second drives Phase 2 path selection per FR-014/FR-016. The two-run requirement applies ONLY to a uniform-`drift_not_reproduced` first run; segment-level Phase 1 outcomes are accepted on one run if they pass the FR-017 cell-agreement gate.
- Q: What scope does Phase 2(a)'s n=100 verification sweep run — the full M6.1 matrix (embed + chat_stream) or chat_stream cells only? → A: Full M6.1 matrix at n=100 (option A). Phase 2(a) MUST run all 6 cells × 3 cohorts × n=100 (1800 RPCs, ~75 min, ~$1.50). The report MUST include BOTH the chat_stream drift-cleared verification (the primary signal) AND an embed regression check (each embed cohort's `engine_forward_ms` mean within 5% of M6.1's published embed mean for that cohort × cell). Rationale: M6.1's published `supersedes_m6_under_enable_prompt_embeds` table includes embed verdicts; if Phase 2(a)'s symmetrisation code change accidentally perturbs the embed measurement window even slightly, those verdicts would silently rest on shifted methodology. Spending ~25 extra minutes and ~$0.50 once to confirm embed cells are unchanged is small insurance against a silent regression. Embed-regression failure (any cohort × cell outside the 5% band) MUST surface as an `embed_regression_warning` flag in the Phase 2(a) report and block the milestone from closing until the operator either (i) identifies and reverts the perturbing change, or (ii) explicitly acknowledges the regression with a justification recorded in `phase_2_choice`.

The acknowledgement path triggers publication of a top-level JSON key `embed_baseline_post_symmetrisation` containing for each of the 3 embed cells × 3 cohorts: `engine_forward_ms` mean + 95% CI half-width, `n_successes`, and a per-entry `regression_warning` flag echoing FR-015b's check. M6.2 anchors against this section for embed cells; M6.1's published embed verdicts remain valid AS PUBLISHED under M6.1's instrumentation (no revocation), and M6.1's JSON additionally gains a per-row methodology supersedence note on the affected `supersedes_m6_under_enable_prompt_embeds` rows pointing at M6.1.1's embed baseline. The sentinel-object schema from round-2 Q1 is extended in parallel: the top-level `embed_baseline_post_symmetrisation` key MUST always be present with shape `{phase_2_path, baseline_source ∈ {"m6_1_1", "m6_1", "not_applicable"}, pointer, cells?}` — `baseline_source = "m6_1_1"` when Phase 2(a) acknowledges or when embed regression passes (cells populated either way for symmetry with chat_stream); `baseline_source = "m6_1"` under Phase 2(b) / `drift_not_reproduced_confirmed`; `baseline_source = "not_applicable"` under `inconclusive`. This rule applies *only* when the operator acknowledges; if the operator reverts the perturbing change instead, FR-015b's regression check re-runs on the revised code and `embed_baseline_post_symmetrisation` is published with `baseline_source = "m6_1_1"` and cells populated with the post-revert (passing) numbers — equivalent to the no-regression case.
- Q: If Phase 1's perturbation budget (FR-012: <500 µs per RPC for the four-checkpoint reads) is exceeded, should the classifier still run? → A: No — block with exit code `4` (option B). The FR-010 classifier compares segment-spread ratios; a perturbation that systematically inflates one segment's measurement biases the ratio and could flip a borderline classification. If the perturbation budget is exceeded on any (cell × cohort), the harness MUST exit with code `4` ("instrumentation perturbation budget exceeded; reduce checkpoint cost and re-run `--m6_1_1-diagnose`") and MUST NOT write classifications, MUST NOT publish a Phase 1 classification table in the markdown report, and MUST NOT permit `--m6_1_1` (Phase 2(a)) to run. The Phase 1 report MAY still be written with the raw multi-point timing table as audit evidence, but the executive summary MUST clearly state "PERTURBATION BUDGET EXCEEDED — classifications not computed; see methodology section". This treats the perturbation budget as a hard correctness gate on par with FR-017 (mixed exit `3`), FR-018 (inconclusive / single-run drift_not_reproduced exit `3`), FR-001 (missing baseline exit `1`), and FR-003 (torch-pin exit `2`).
- Q: If the second `--m6_1_1-diagnose` run still returns mixed or inconclusive classifications across the three chat_stream cells, is heterogeneous Phase 2 (per-cell fix + per-cell doc inside one milestone) allowed? → A: No — disallow heterogeneous Phase 2; require milestone split (option B). Mixing a code change for one cell with a doc-only outcome for another violates the one-variable-per-milestone Phase Discipline that motivated M6.1.1 to exist as its own narrow milestone. After a confirming second `--m6_1_1-diagnose` run that leaves classifications still-divergent (mixed segment-level labels) or still-inconclusive on any cell, the harness MUST write the M6.1.1 report with `phase_2_path = "split_required"` and exit code `5` ("milestone split required — open successor sub-milestones M6.1.1a / M6.1.1b before any code or doc change"). The Phase 1 report is preserved as evidence; `phase_2_choice` records the per-cell divergence pattern and a one-line operator note describing the proposed split. M6.1.1 itself closes without a Phase 2 outcome; M6.2 cannot proceed (its SC-008 anchor would be `baseline_source = "not_applicable"`) until at least one successor sub-milestone publishes. Acceptable successor splits: M6.1.1a (instrumentation_artifact cells → fix path) + M6.1.1b (channel_dependent_batching cells → doc path); OR M6.1.1a (segment-level cells → whichever fix/doc) + M6.1.1c (re-run Phase 1 with expanded checkpoints to resolve inconclusive). The shape of the split is operator-discretion but MUST be one-variable-per-successor.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Diagnose the chat_stream engine_ttft spread via multi-point timing (Priority: P1)

The operator runs a focused mini-sweep that emits four timestamps per chat_stream RPC from both REST and gRPC paths — at handler entry, just before `engine.generate(...)`, at the first streamed chunk, and at terminal emit — so the published M6.1 `engine_cost_drift_warning` can be attributed to either measurement-window asymmetry (instrumentation artifact) or channel-dependent batching (real engine effect). Without this attribution, M6.2 cannot use the M6.1 verdicts as a null anchor and every downstream milestone's per-cohort engine-cost decomposition carries an unresolved 14–17% asterisk.

**Why this priority**: This is the entire milestone's reason for existing. Without Phase 1's diagnostic data, Phase 2 cannot be chosen, M6.1's published table cannot be defended past the next sweep, and M6.2 cannot start. Every other user story in this spec is downstream of Phase 1's classification output.

**Independent Test**: Operator runs `python -m vllm_grpc_bench --m6_1_1-diagnose --m6_1_1-modal-region=eu-west-1` against the matrix M6.1 used (6 cells × 3 cohorts × n=50). The harness publishes `docs/benchmarks/m6_1_1-engine-cost-instrumentation.{md,json}` containing: (a) a per-cohort multi-point timing table for each of the 6 cells (the three embed cells included as controls; per FR-031 / Out of Scope, drift attribution is computed only for chat_stream), (b) per-segment delta analysis (a→b, b→c, c→d) per cohort per chat_stream cell, and (c) one of four classifications per chat_stream cell: `instrumentation_artifact` / `channel_dependent_batching` / `inconclusive` / `drift_not_reproduced`. The operator can read the classification and the segment carrying the spread without running anything further.

**Acceptance Scenarios**:

1. **Given** M6.1's published JSON exists at the default baseline path, **When** the operator runs `--m6_1_1-diagnose` against an M6.1.1-prepared Modal deployment, **Then** the harness publishes a markdown + JSON report containing 4-checkpoint timings per cohort per cell, per-segment deltas, and one of {`instrumentation_artifact`, `channel_dependent_batching`, `inconclusive`, `drift_not_reproduced`} for each of the 3 chat_stream cells.
2. **Given** M6.1's published JSON is missing or unreadable, **When** the operator runs `--m6_1_1-diagnose`, **Then** the harness exits with code `1` and prints a stderr message naming the missing baseline file and the `--m6_1_1-m6-1-baseline` flag for overriding it.
3. **Given** the operator's local `torch` version does not match the pin enforced in M6.1, **When** the operator runs `--m6_1_1-diagnose`, **Then** the harness exits with code `2` BEFORE attempting any Modal deployment, identical to M6.1's FR-006 gate, and prints the expected vs actual version.

### User Story 2 — Symmetrise the engine_ttft measurement window OR publish channel-dependent batching as a finding (Priority: P2)

Based on Phase 1's classification, the operator either (a) applies a code fix that moves the REST or gRPC bracketing so both paths capture the same engine work, then runs an n=100 verification sweep against the M6.1 matrix to confirm the drift warning clears; or (b) treats the spread as a real channel-dependent batching effect, updates `contracts/instrumentation.md` with an operator-facing interpretation, and annotates M6.1's published JSON with a methodology supersedence pointer to M6.1.1's diagnosis. Path choice is dictated by Phase 1's data, never pre-committed.

**Why this priority**: Phase 2 turns Phase 1's diagnosis into a closed methodology gap. Without Phase 2, M6.1's published table still carries the drift asterisk forever; M6.2 still cannot use M6.1 as a null anchor. P2 (not P1) because Phase 2 cannot run until Phase 1 finishes — strict ordering — but they are part of the same milestone.

**Independent Test**: If Phase 1 returned `instrumentation_artifact`, the operator applies the symmetrisation code change to the path Phase 1 identified, runs `python -m vllm_grpc_bench --m6_1_1 --m6_1_1-modal-region=eu-west-1`, and the harness publishes a Phase-2(a) verification sweep that shows `engine_cost_drift_warning=false` across all three chat_stream cells (each cohort's `engine_ttft_ms` mean within 5% of the unweighted cohort-average for that cell). If Phase 1 returned `channel_dependent_batching`, the operator runs a doc-only commit updating `contracts/instrumentation.md` and the M6.1.1 markdown report; the M6.1.1 JSON records the chosen path so M6.2 (which reads it as a hard precondition) can interpret M6.1's per-cohort engine_ttft correctly.

**Acceptance Scenarios**:

1. **Given** Phase 1 classified all three chat_stream cells as `instrumentation_artifact` and a code change has symmetrised the bracketed window, **When** the operator runs `--m6_1_1` (Phase 2(a) verification sweep, n=100 across the full 6-cell M6.1 matrix), **Then** the harness publishes a verification report showing (i) `engine_cost_drift_warning=false` for all three chat_stream cells with each cohort's `engine_ttft_ms` mean within 5% of the unweighted cohort-average; (ii) a `chat_stream_baseline_post_symmetrisation` section with the 9 (cell × cohort) entries per FR-015a; (iii) an embed regression check passing per FR-015b (each embed cohort's `engine_forward_ms` mean within 5% of M6.1's published mean); AND M6.1's published JSON is annotated with a forward pointer to the M6.1.1 verification report.
2. **Given** Phase 1 classified the chat_stream drift as `channel_dependent_batching`, **When** the operator updates `contracts/instrumentation.md` with a section explaining the effect and the operator-facing interpretation of per-cohort `engine_ttft_ms` differences, **Then** the M6.1.1 markdown report records the doc-only Phase 2(b) outcome, M6.1's published JSON is annotated with a forward pointer naming the channel-dependent-batching finding, and no verification sweep is run.
3. **Given** Phase 1 returned `inconclusive` on any cell, **When** the harness completes, **Then** it exits with code `3` and a stderr message instructing the operator to re-run `--m6_1_1-diagnose` per FR-018; the harness does NOT auto-proceed to Phase 2.
4. **Given** Phase 1 returned uniform `drift_not_reproduced` across all three chat_stream cells on the first `--m6_1_1-diagnose` run, **When** the harness completes, **Then** it exits with code `3` per FR-018 instructing the operator to re-run `--m6_1_1-diagnose` to confirm; only after a second confirming run does the harness write the M6.1.1 report with `phase_2_path = "drift_not_reproduced_confirmed"` and annotate M6.1's JSON.
5. **Given** Phase 1 returned mixed classifications across the three chat_stream cells (e.g., two `instrumentation_artifact`, one `channel_dependent_batching`), **When** the harness completes, **Then** it exits with code `3` per FR-017 naming the divergent cells; on the second `--m6_1_1-diagnose` run, if classifications now agree the harness proceeds with the agreed Phase 2 path, but if classifications still diverge the harness writes `phase_2_path = "split_required"` and exits code `5` (heterogeneous Phase 2 inside M6.1.1 is disallowed; operator opens successor sub-milestones per FR-017(b)).

### Edge Cases

- **Phase 1 mini-sweep produces a 4-segment timing in which the spread is concentrated in `c→d` (post-engine, pre-emit) on one path only** — not anticipated by the (a) measurement-window-asymmetry / (b) channel-dependent-batching dichotomy. Handling: classifier returns `inconclusive` (neither `seg_ab` nor `seg_bc` meets the 0.80 magnitude threshold); harness exits code `3` per FR-018; on re-run, if `inconclusive` persists, the operator chooses between extending the checkpoint set (additional spec round) or proceeding with heterogeneous Phase 2 per FR-017(b).
- **Phase 1 mini-sweep itself shows no per-cohort spread above 5% on chat_stream cells** (M6.1's reported drift was a transient infrastructure artifact rather than a persistent issue). Handling: classifier returns `drift_not_reproduced`; the harness MUST refuse to close M6.1.1 on a single run (exit code `3` per FR-018); only after a confirming second run is `phase_2_path = "drift_not_reproduced_confirmed"` written, M6.1's flag is preserved as published, and the `methodology_supersedence` note is appended to M6.1's JSON.
- **M6.1's published JSON is moved, renamed, or unreadable when M6.1.1 starts.** Handling: hard precondition; harness exits with code `1` and instructs the operator to pass `--m6_1_1-m6-1-baseline=<path>` explicitly.
- **The four timing checkpoints themselves perturb engine_ttft enough to shift the (b→c) measurement vs M6.1's per-cohort baseline.** Handling: instrumentation is timestamp-recording only (`perf_counter_ns()` on the local process, no network I/O at the checkpoints, no logging); perturbation budgeted at <500 µs per checkpoint and asserted in the Phase 1 report's methodology note. If the budget is exceeded a stderr warning fires and the operator is asked to weight the Phase 1 numbers cautiously.
- **gRPC trailing metadata size grows from the 4 new keys enough to affect protocol overhead vs M6.1.** Handling: 4 keys × ≤20 bytes each ≈ 80 bytes of additional metadata, well below the ~1 KB metadata budget M6 already operates within. No mitigation expected; report's methodology section records the actual increase in bytes for audit.
- **The REST shim's SSE terminal event already carries some per-RPC instrumentation fields (M6's `engine_ttft_ms`, `engine_tpot_ms`); adding four more JSON fields could collide with existing keys.** Handling: the four new keys are namespaced under a `m6_1_1_timings` sub-object on the terminal SSE JSON event so the existing M6 field set remains exactly as it was; the namespace is recorded in `contracts/instrumentation.md`.
- **gRPC chat_stream trailing metadata already carries the M6 `engine_ttft_ms` and `engine_tpot_ms` keys; the four new keys could collide.** Handling: new keys are prefixed `m6_1_1_t_` (e.g., `m6_1_1_t_handler_entry`, `m6_1_1_t_pre_engine`); the prefix is recorded in `contracts/instrumentation.md`.
- **Phase 2(a) verification sweep also fires `chat_stream_control_drift_warning` (FR-029 from M6.1)** because the symmetrised instrumentation changes the measurement window enough that M6.1's published chat_stream CI no longer overlaps the new one. Handling: expected and documented as methodology-only (the brackets changed, not the engine). Phase 2(a) MUST publish a fresh chat_stream baseline (`chat_stream_baseline_post_symmetrisation`, see FR-015a) so downstream milestones (M6.2 and beyond) have a usable null-anchor reference under the new instrumentation. M6.1's chat_stream verdicts remain valid AS PUBLISHED under M6.1's instrumentation; no verdict revocation. The verification report annotates the control-drift flag with "expected — reflects bracketing change in Phase 2(a) symmetrisation; not infrastructure drift".
- **The drift warning fires only on a subset of the three chat_stream cells when re-measured (e.g., c=8 fires `instrumentation_artifact` but c=1/c=4 fire `drift_not_reproduced` or `channel_dependent_batching`).** Handling: per-cell classification (FR-010); harness applies the FR-017 divergence rule — if classifications from the segment-level set `{instrumentation_artifact, channel_dependent_batching, inconclusive}` disagree across cells, exit code `3` and operator re-runs; `drift_not_reproduced` cells are excluded from the divergence test. After a confirming second run with still-divergent classifications, the harness writes `phase_2_path = "split_required"` and exits code `5` — heterogeneous Phase 2 inside M6.1.1 is disallowed (one-variable-per-milestone discipline); the operator opens successor sub-milestones per FR-017(b).

### Functional Requirements

If the file is missing, unreadable, or fails schema validation against `schema_version=="m6_1.v1"`, the harness MUST exit with code `1` and a stderr message naming the file and the override flag. The check MUST run BEFORE any Modal deployment is initiated.

The harness MUST emit a methodology section in the published report confirming each value matches M6.1's published `run_meta`.

#### Phase 1 — Diagnosis Mini-Sweep

- **FR-005**: Phase 1 MUST run 6 cells × 3 cohorts × n=50 measurement RPCs (900 total) — identical cell shape to M6.1, half the RPC count to keep wall-clock under ~30 min. Cells are `embed × {c=1, c=4, c=8}` and `chat_stream × {c=1, c=4, c=8}` at `hidden_size=4096`.
- **FR-006**: Phase 1 MUST emit four timing checkpoints on each chat_stream RPC, from both REST and gRPC paths: (a) `handler_entry` — the FastAPI handler / gRPC servicer receives the request; (b) `pre_engine` — just before `engine.generate(...)` is invoked; (c) `first_chunk` — the first streamed output chunk is produced; (d) `terminal_emit` — the terminal SSE event is emitted (REST) / trailing metadata is sent (gRPC). Each checkpoint is a `perf_counter_ns()` reading from the same local process clock.
- **FR-007**: REST chat_stream MUST emit the four timestamps as fields on a `m6_1_1_timings` sub-object on the terminal SSE event's JSON payload. The existing M6 SSE field set MUST be preserved exactly (no key changes, no order changes). The four fields are: `handler_entry_ns`, `pre_engine_ns`, `first_chunk_ns`, `terminal_emit_ns`.
- **FR-008**: gRPC chat_stream MUST emit the four timestamps as additional trailing-metadata keys, prefixed `m6_1_1_t_`: `m6_1_1_t_handler_entry`, `m6_1_1_t_pre_engine`, `m6_1_1_t_first_chunk`, `m6_1_1_t_terminal_emit`. Existing M6 trailing-metadata keys MUST be preserved unchanged.
- **FR-009**: Phase 1 MUST compute three per-segment deltas per RPC: `seg_ab_ms = (pre_engine − handler_entry) × 1e-6`, `seg_bc_ms = (first_chunk − pre_engine) × 1e-6`, `seg_cd_ms = (terminal_emit − first_chunk) × 1e-6`. Each segment is aggregated per cohort per cell (mean + 95% CI half-width via bootstrap n_boot=10_000, matching M6.1's confidence-interval methodology).
- **FR-010**: Phase 1 MUST classify each chat_stream cell with exactly one of four labels, deterministically derived from the per-segment data via the **magnitude-equivalence classifier**. For each cell, compute the per-cohort means of `engine_ttft_ms`, `seg_ab_ms`, `seg_bc_ms` (n=50 per cohort, bootstrap CI as in FR-009), and define `spread(x) = max(x) − min(x)` over the three cohort means. Then:
  - `drift_not_reproduced` — if `spread(engine_ttft_ms) / mean(engine_ttft_ms) < 0.05` at the cell level. M6.1's reported drift is not reproduced at n=50; subsequent classification skipped for this cell.
  - `instrumentation_artifact` — else if `spread(seg_ab_ms) / spread(engine_ttft_ms) ≥ 0.80`. The chat_stream `engine_ttft_ms` spread is mostly explained (≥80%) by spread in the pre-engine bracket (one transport path includes work the other does not).
  - `channel_dependent_batching` — else if `spread(seg_bc_ms) / spread(engine_ttft_ms) ≥ 0.80`. The spread lives inside the engine itself (different first-token latencies by cohort — real continuous-batching effect).
  - `inconclusive` — else (the spread is distributed across multiple segments without one dominating, or the data exhibits anomalies such as negative spread ratios from non-monotonic cohort ordering).

  Classifier evaluation order is deterministic and `drift_not_reproduced` is checked first: a cell whose Phase 1 spread is below 5% short-circuits the segment-level checks (the segment ratios are undefined when the denominator is near zero). The 5% drift_not_reproduced threshold and the 0.80 attribution threshold are spec-level constants documented in this FR and reflected verbatim in the published Phase 1 markdown so the operator can reproduce any classification by hand from the multi-point timing table.
- **FR-011**: Phase 1 MUST also collect the four timestamps on embed cells as a control (no per-segment drift expected; M6.1's `engine_cost_drift_warning` did not fire on embed cells). The data is published in the Phase 1 report for audit but is NOT used to drive classification or the Phase 2 path decision.
- **FR-012**: Phase 1 MUST verify the instrumentation perturbation budget: total time spent in the four checkpoint reads per RPC MUST be less than 500 µs on average (per cohort per cell). If exceeded on any (cohort, cell) pair, the harness MUST exit with code `4` ("instrumentation perturbation budget exceeded") and MUST NOT compute FR-010 classifications (segment-ratio bias from perturbation could flip a borderline classification), MUST NOT publish a Phase 1 classification table in the markdown executive summary, and MUST NOT allow `--m6_1_1` (Phase 2(a)) to run on that Phase 1 data. The Phase 1 markdown report MAY still be written with the raw multi-point timing table as audit evidence, but the executive summary MUST start with "PERTURBATION BUDGET EXCEEDED — classifications not computed; see methodology section". Operator action: reduce checkpoint cost (e.g., switch from `perf_counter_ns()` to a cheaper monotonic clock; eliminate intermediate variables; inline the timestamp capture) and re-run `--m6_1_1-diagnose`. The `instrumentation_perturbation_warning` flag is still emitted in the JSON for audit, but it is a *gate* now, not a soft warning.
- **FR-013**: Phase 1 MUST be driven by the CLI flag `--m6_1_1-diagnose` (mutually exclusive with `--m6_1_1`, `--m6_1`, `--m6_1-smoke`, `--m6`, `--m6-smoke`, `--m5_2`, `--m5_1`, `--m5`, `--m4`, `--m3`). No M6.1.1 smoke flag exists per FR-028; the n=50 mini-sweep is itself the smoke-equivalent wiring check.

### Key Entities

- **M6.1.1 Phase 1 Diagnostic Run**: The result of one `--m6_1_1-diagnose` invocation. Contains 4-checkpoint timing data per cohort per cell (with embed cells as audit-only controls), per-segment delta aggregates, per-chat_stream-cell classification, and the perturbation-budget audit. Persisted to the JSON companion under `multi_point_timings` and `phase_1_classifications`.
- **Timing Checkpoint**: One of four named per-RPC timestamps (`handler_entry`, `pre_engine`, `first_chunk`, `terminal_emit`). Recorded as `perf_counter_ns()` on the local server process. Emitted on the wire (REST: `m6_1_1_timings` sub-object on terminal SSE event; gRPC: `m6_1_1_t_*` trailing metadata keys).
- **Per-Segment Delta**: One of three named per-RPC durations (`seg_ab_ms`, `seg_bc_ms`, `seg_cd_ms`), each derived from a consecutive checkpoint pair. Aggregated per cohort per cell.
- **Phase 1 Classification**: A per-cell label in `{instrumentation_artifact, channel_dependent_batching, inconclusive, drift_not_reproduced}`. Determined deterministically from per-segment data (FR-010 thresholds). Drives Phase 2 path selection (or the decision NOT to run Phase 2).
- **Phase 2(a) Verification Run**: The result of one `--m6_1_1` invocation, run after a symmetrisation code fix has been applied. Full M6.1 matrix at n=100 (6 cells × 3 cohorts = 1800 RPCs). Publishes drift-cleared/persistent status per chat_stream cell, fresh chat_stream baseline (FR-015a), fresh embed baseline (FR-015c), and embed regression check pass/fail per cell (FR-015b).
- **Phase 2(b) Documentation Outcome**: A doc-only result containing the path to the updated `contracts/instrumentation.md` and a summary of the channel-dependent-batching finding. No verification sweep runs.
- **Methodology Supersedence Note**: An additive annotation written to both M6.1's JSON (top-level `methodology_supersedence` key) and M6.1's markdown (one-line forward pointer) directing readers to M6.1.1's published report. Committed by the same PR that publishes M6.1.1.

## Assumptions

- M6.1's published JSON (`docs/benchmarks/m6_1-real-prompt-embeds.json` at the time of M6.1.1's spec drafting) is the immutable input for comparison; per-cohort `engine_ttft_ms` means and CIs in that file are the baseline M6.1.1 diagnoses against. M6.1.1 adds the `methodology_supersedence` annotation but does not modify any other field.
- Modal A10G `eu-west-1` availability and scheduler behavior is approximately stable run-to-run (M6 and M6.1 used the same region and reported consistent results). If a Phase 1 mini-sweep shows wildly different cohort means from M6.1's, the most likely explanation is M6.1.1's instrumentation perturbation, not infrastructure drift; the perturbation budget audit (FR-012) addresses this.
- vLLM 0.20.1's continuous batching behavior at `max_tokens=50`, `c ∈ {1, 4, 8}` is approximately deterministic enough that the per-cohort `engine_ttft_ms` spread observed in M6.1 reproduces at n=50 (Phase 1's sample size) above the `drift_not_reproduced` threshold. If it does not, that itself is the finding (`drift_not_reproduced` classification).
- The four timing checkpoints are descriptive of the `engine_ttft_ms` window in both REST and gRPC paths at a level sufficient to localize the M6.1 spread to one segment. If implementation discovers that a path's internal flow needs a fifth checkpoint, the additional checkpoint is added during `/speckit-plan` or `/speckit-tasks` with the wire-emission contract growing to match (the 4-key minimum in FR-007 / FR-008 is a floor, not a ceiling).
- The two-hypothesis dichotomy (`instrumentation_artifact` vs `channel_dependent_batching`) is the operationally relevant outcome dimension; `inconclusive` and `drift_not_reproduced` are documented fallbacks but the project expectation is that Phase 1 returns one of the two primary classifications on at least 2 of the 3 chat_stream cells.
- M6.1's `chat_stream_control_drift_warning` (FR-029 in M6.1) is unrelated to `engine_cost_drift_warning` (FR-022 in M6.1) — they detect different things. M6.1.1 only addresses `engine_cost_drift_warning`; the `chat_stream_control_drift_warning` flag's mechanism remains as M6.1 defined it. (Phase 2(a) verification may incidentally fire it; the report explicitly distinguishes the two flags.)
- `torch==2.11.0` continues to be the right client-side pin (M6.1's FR-006). If the project updates the pin between M6.1 and M6.1.1 publishing, M6.1.1's pin gate is updated to match in the same PR.
- The default Modal region (`eu-west-1`) is the right region for the verification sweep to match M6.1's network-stack characteristics. Switching regions between M6.1 and M6.1.1 would itself introduce a confounder; the spec defaults match M6.1 deliberately.
- The operator runs Phase 1, inspects the report, manually applies any code change (Phase 2(a)) or doc change (Phase 2(b)), and then runs Phase 2(a) verification if applicable. The harness does NOT auto-edit code or auto-apply doc changes; the human-in-the-loop step between Phase 1 and Phase 2 is intentional (the diagnosis informs which path to take, and the operator commits to it).
