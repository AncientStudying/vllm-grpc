# Implementation Plan: M4 — Time-Axis Channel & Schema Tuning

**Branch**: `016-m4-time-axis-tuning` | **Date**: 2026-05-10 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/016-m4-time-axis-tuning/spec.md`

## Summary

M4 closes the methodology gap M3 documented in its published time report. M3 produced defensible bytes verdicts but left every chat_stream total wall-clock cell `noise_bounded` because the mock engine's deterministic per-token pacing dominates streaming wall-clock and per-axis fresh M1_BASELINE cohorts drifted ~10–15% across batches on macOS. M4 fixes both upstream causes — adds a no-pacing mode to the mock engine and a shared-baseline mode to the orchestrator — promotes TTFT to a first-class verdict metric, then re-runs the four-axis channel sweep (`max_message_size`, keepalive, compression, HTTP/2 framing) at hidden_size 2048 / 4096 / 8192 against that shared baseline. The deferred-from-M3 schema candidates (packed scalars on token-id, `oneof` flattening, alternative streaming chunk granularity) are then measured against per-path frozen-channel baselines built from the channel-sweep winners. M3's bytes report stays in place; the M4 report supersedes M3's `noise_bounded` time cells and is published as a sibling at `docs/benchmarks/m4-time-axis-tuning.{md,json}` with a JSON schema that is a **strict superset** of M3's `m3-channel-tuning-time.json` (additive only — M3 readers continue to work).

The technical approach reuses the M3 harness modules under `tools/benchmark/src/vllm_grpc_bench/`. `mock_engine.py` gains a `pace_tokens: bool` flag; the orchestrator (`m3_sweep.py`, renamed-or-extended via a sibling `m4_sweep.py`) gains a shared-baseline orchestration mode and a per-path frozen-channel baseline builder; the recommendation builder is extended so TTFT is the primary chat_stream metric (no longer a re-analysis-only path) and so cohorts whose dominant per-RPC cost is client-side are tagged `client_bound` and excluded from `recommend` tallies; candidate cohorts default to n ≥ 100 with a borderline-expand step to n ≥ 250 when the candidate's 95% CI touches the baseline cohort's 95% CI. The CLI gains `--m4`, `--no-pacing`, and `--shared-baseline` flags. Schema candidates are evaluated by introducing isolated `.proto` candidate variants under `proto/vllm_grpc/v1/m4-candidates/` (regenerated via `make proto` per Constitution I) — production proto stays untouched until a candidate is explicitly accepted in a follow-up change.

## Technical Context

**Language/Version**: Python 3.12 (`requires-python = ">=3.12,<3.13"` in `pyproject.toml`) — unchanged from M3.
**Primary Dependencies**: `grpcio==1.80.0` (pinned in `[dependency-groups] graph-targets`), `vllm` 0.20.1 on Linux / `vllm-metal` 0.2.0 on macOS (still type-shape parity only; mock engine drives the wire), `protobuf` (transitive via `grpcio-tools`), `FastAPI` (proxy), `httpx` (bench client), `numpy` (mock embedding tensors and CI math). No new runtime dependencies.
**Storage**: N/A on the runtime path. M4 report lands as JSON+Markdown under `docs/benchmarks/m4-time-axis-tuning.{md,json}` (committed); transient per-iteration timing arrays land under `bench-results/m4-full/` (gitignored, same convention as M3).
**Testing**: `pytest` for unit and integration tests (`make check`). New M4 tests follow the M3 pattern: harness unit tests under `tools/benchmark/tests/`, proto-candidate compile checks under existing `make proto` plumbing, integration tests under `tests/integration/` covering the CLI-driven `--m4` mode.
**Target Platform**: macOS (Apple Silicon M2/M3) and Linux x86-64, CPU-only. Mock model has no GPU dependency. Cross-host transport is **out of scope** for M4 unless a specific axis's loopback-masked verdict justifies it (per spec Assumptions and FR-010).
**Project Type**: Python `uv` workspace with multiple packages — `packages/{client,frontend,gen,proxy}` and `tools/benchmark`. M4 changes concentrate in `tools/benchmark/`; surgical changes to `proto/vllm_grpc/v1/m4-candidates/` for US3.
**Performance Goals**: M4 *establishes* time-axis numbers; it does not preset a target. The empirical questions (per spec SC-001..SC-006): which channel-axis settings reduce TTFT (chat_stream) or total per-RPC wall-clock (embed) at hidden_size 2048 / 4096 / 8192 with 95% CIs that strictly clear the shared-baseline; which deferred-from-M3 schema candidates show measurable wins on bytes or time at hidden_size 4096 (4096-first cascade, expand on positive signal).
**Constraints**:
- Constitution I (Proto-First): schema candidates MUST be `.proto` edits with stubs regenerated by `make proto`. No hand-written equivalents. Candidate proto variants live in a sibling namespace until adopted.
- Constitution V (Honest Measurement): `client_bound` cohorts excluded from `recommend` tallies; loopback caveat note attached to physically-loopback-dependent axes (keepalive, HTTP/2 framing) every time; negative-result schema candidates appendixed (FR-014).
- Spec FR-002: shared-baseline cohort n ≥ 100; candidate cohorts default n ≥ 100, expand to n ≥ 250 on borderline (CIs touch).
- Spec FR-005: harness records within-cohort CV (verdict metric) on every cohort and surfaces it in the published report. The run completes regardless of CV; a configurable warn threshold (`--baseline-cv-warn`, default `0.05`) only controls a closing warning and a `noisy_baseline` flag. Verdict trust adjudication is the reader's job.
- Spec FR-007: every M3 `noise_bounded` cell must have a non-`noise_bounded` M4 verdict; the `noise_bounded` literal MUST NOT appear in the M4 report.
- Spec FR-015: M4 JSON schema is a strict superset of M3's `m3-channel-tuning-time.json` — additive only.
**Scale/Scope**:
- 2 shared M1_BASELINE cohorts (one per path) at n ≥ 100 = up front.
- 4 axes × 3 widths × 2 paths × ≥2 configs per axis = ~48 candidate cells (US2). Borderline-expand may grow some to n ≥ 250.
- 2 per-path frozen-channel baseline cohorts at n ≥ 100 (US3 prerequisite).
- 3 named schema candidates (FR-012) × at least hidden_size 4096 = 3+ cohorts (cascade adds 2048 + 8192 for `recommend`/borderline candidates).
- Total expected M4 runtime budget: CPU-only, target ≤ 4 hours for the full sweep on a single commodity host (matches M3's budget).
