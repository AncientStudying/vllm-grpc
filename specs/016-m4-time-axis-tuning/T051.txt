  T051 — Run the full M4 sweep

  How long it takes

  Plan budget is ≤ 4 hours of mostly-idle CPU. Practical range on a quiet host: 2–4 hours.

  Pre-flight (5 min)

  # 1. Clean tree on the M4 branch.
  git status

  # 2. Regenerate stubs (cheap; needed if anyone touched .proto since last build).
  make proto

  # 3. Confirm candidate stubs import.
  uv run python -c "from vllm_grpc.v1.m4_candidates import packed_token_ids_pb2; print('ok')"

  # 4. Quiet the host. Quit Slack, Chrome, Docker Desktop, Spotify, etc.
  #    Background load drives within-cohort CV. The run still completes regardless of CV
  #    (FR-005 record-and-report model — see research.md R-11), but a quiet host produces
  #    tighter CIs and fewer cohorts flagged `noisy_baseline=true`.

  Trigger the full sweep

  Recommended:

  mkdir -p logs
  uv run python -m vllm_grpc_bench --m4 \
    2>&1 | tee logs/m4-full-$(date +%Y%m%d-%H%M%S).log

  Defaults that apply: --no-pacing --shared-baseline --baseline-n=100 --candidate-n=100
  --expand-n=250 --warmup-n=10 --baseline-cv-warn=0.05
  --widths=2048,4096,8192 --paths=embed,chat_stream
  --axes=max_message_size,keepalive,compression,http2_framing.

  Backgrounded (so you can step away):

  mkdir -p logs
  nohup uv run python -m vllm_grpc_bench --m4 \
    > logs/m4-full-$(date +%Y%m%d-%H%M%S).log 2>&1 &
  echo $! > logs/m4-full.pid

  Monitor

  # Live cohort-by-cohort progress.
  tail -f logs/m4-full-*.log

  # Count completed cohorts (each prints "[N/total] cell=... verdict=...").
  grep -c "^\[" logs/m4-full-*.log

  # Process still alive?
  kill -0 "$(cat logs/m4-full.pid)" 2>/dev/null && echo running || echo exited

  # Transient per-iteration JSON landing under bench-results/m4-full/.
  ls -lt bench-results/m4-full/ 2>/dev/null | head

  When complete, stdout shows:

  M4 sweep complete: <N> recommend, <M> no_winner, <K> client_bound. <P> M3 cells superseded.

  and docs/benchmarks/m4-time-axis-tuning.{md,json} materializes. If any baseline cohort's
  CV exceeded --baseline-cv-warn, a closing stderr warning lists the offending cohort ids;
  the JSON entries also carry `noisy_baseline: true` for downstream filtering.

  If the run aborts

  Exit: 2
  Meaning: CLI flag/arithmetic violation
  Recovery: Fix the flag named in stderr
  ────────────────────────────────────────
  Exit: 4
  Meaning: Internal invariant failure
  Recovery: File a bug — should never happen
  ────────────────────────────────────────
  Exit: 5
  Meaning: Candidate proto compilation failure
  Recovery: make proto standalone, fix the offending .proto

  Exit 3 is no longer used. Per FR-005 (research.md R-11), the harness records within-cohort
  CV per cohort and never aborts on it. If you see noisy baselines in the report, re-run
  only those cohorts in a follow-up — the previous sweep's data is still on disk.

  Spot-check (T051 acceptance)

  # 1. No noise_bounded leaked into the M4 report (FR-007).
  jq '[.recommendations[].verdict] | unique' docs/benchmarks/m4-time-axis-tuning.json

  # 2. M3-shape readers still parse the JSON (FR-015 strict-superset).
  jq '{mode, axes, widths, paths, iterations_per_cell, seed,
       p2_revision, frozen_channel, cohort_count: (.cohorts | length)}' \
     docs/benchmarks/m4-time-axis-tuning.json

  # 3. Supersession entries cover the M3 noise_bounded cells.
  jq '.supersedes | length' docs/benchmarks/m4-time-axis-tuning.json

  # 4. Loopback caveat axes — should equal {keepalive, http2_framing} on single-host.
  jq '.loopback_caveat_axes' docs/benchmarks/m4-time-axis-tuning.json

  # 5. Borderline-expanded cohorts (the n=100→250 cascade fired here).
  jq '.cohorts[] | select(.expansion_record.expanded == true) | .cell_id' \
     docs/benchmarks/m4-time-axis-tuning.json

  # 6. client_bound cohorts (excluded from recommend tallies, FR-004).
  jq '.cohorts[] | select(.client_bound == true) | .cell_id' \
     docs/benchmarks/m4-time-axis-tuning.json

  # 7. Per-cohort CV — surfaces noisy cohorts for adjudication (FR-005).
  jq '.cohorts[] | {cell_id, time_cv, ttft_cv, noisy_baseline}' \
     docs/benchmarks/m4-time-axis-tuning.json

  # 8. Just the noisy baselines.
  jq '.cohorts[] | select(.noisy_baseline == true) | {cell_id, time_cv, ttft_cv}' \
     docs/benchmarks/m4-time-axis-tuning.json

  Then T052/T053 (cheap)

  # T052 — add seed + host details to the markdown's methodology preamble,
  # then commit the published reports.
  git add docs/benchmarks/m4-time-axis-tuning.md docs/benchmarks/m4-time-axis-tuning.json
  git commit -m "[Spec Kit] Publish M4 report from validated sweep: 016-m4-time-axis-tuning"

  # T053 — flip docs/PLAN.md M4 line from "active" to "delivered" and
  # cross-link the report from the milestone overview.

  uv run python -m vllm_grpc_bench --m4 \
    --skip-schema --widths=4096 --paths=embed --axes=compression
