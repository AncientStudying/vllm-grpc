# Feature Specification: M5.1 — REST vs gRPC Head-to-Head on Real Wire

**Feature Branch**: `018-m5-1-rest-vs-grpc`
**Created**: 2026-05-11
**Status**: Draft
**Input**: User description: "M5.1 — REST vs gRPC Head-to-Head on Real Wire. Please make sure a requirement gets into the specifications to update README.md to current state of project just before PR"

## Background

M5 (delivered 2026-05-11, PR #21) showed that *tuned* gRPC channels deliver substantial wall-clock wins over *default* gRPC on real wire — five winning channel configurations at `embed/h2048` with deltas of −23% to −25% against the M5 cross-host shared baseline, measured against a Modal-hosted MockEngine across a 52 ms median RTT. M5 also resolved the loopback caveat that M4 had attached to the `keepalive` and `http2_framing` axes.

What M5 did **not** do is re-compare REST against those same conditions. The project's cumulative evidence for the headline claim — "tuned protobuf+gRPC is more time-efficient than REST for vLLM interaction" — currently rests on:

- **Bandwidth half**: M1's structural wire-size measurements (chat response 611 B → 65 B, an 89% reduction; embed request 25% reduction from base64 elimination). These are structurally driven by encoding choice (proto bytes vs JSON+base64) and are immune to network conditions, so the bandwidth half of the headline is well-supported and does not require M5.1 to re-prove.
- **Time half**: M1's REST-vs-default-config-gRPC numbers measured over short-haul Modal (single-region A10G) produced **mixed results** — REST faster at c=1 small-body chat (channel-setup overhead dominates the gRPC leg), gRPC faster on the completions wire. M1 never compared *tuned* gRPC against REST, and M5's wins were measured against tuned-gRPC's own default-gRPC baseline, not against REST. The time half of the headline therefore lacks a clean cross-host tuned-gRPC-vs-REST control.

M5.1 closes that gap. It reuses M5's cross-host harness end-to-end (Modal-hosted MockEngine, cross-region RTT, per-cohort RTT probe, `server_bound` classifier, borderline-expand cascade, no methodological change) and adds a REST cohort driven through a FastAPI shim against the same `MockEngine` so the engine variable is held constant between the two protocols. The trade-off — numbers will not be production-quoted absolutes because MockEngine does not exercise a real model — is accepted because M5.1's question is **protocol comparison**, not absolute production performance. M7 (model expansion) is the milestone that re-validates the M5.1 findings against real vLLM.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Tuned-gRPC vs REST Head-to-Head on Real Wire (Priority: P1)

A maintainer needs a single artifact that answers, with cross-host real-wire evidence and the same statistical bar M5 uses (95% CI strictly clears the shared baseline), the question "is tuned protobuf+gRPC time-efficient against REST?" — broken down by access path (chat_stream, embed), hidden_size (2048, 4096, 8192), and concurrency (matching M1's c=1, c=4, c=8). The comparison holds the engine variable constant: both protocols hit the same Modal-hosted `MockEngine` in no-pacing mode, the only operative difference between a REST cohort and a gRPC cohort being the wire protocol and the channel configuration.

**Why this priority**: This is the headline value of M5.1 and the only reason M5.1 exists. Until this lands, the project's time-half claim ("tuned protobuf+gRPC is more time-efficient than REST") cannot be cited cleanly — M1's mixed numbers were measured against default-config gRPC under loopback-ish conditions, and M5's wins are protocol-internal (tuned-gRPC vs default-gRPC). The cross-host head-to-head is the missing measurement in the milestone roadmap.

**Independent Test**: A reviewer reads `docs/benchmarks/m5_1-rest-vs-grpc.md` and the M5.1 JSON file and confirms (a) every (path × hidden_size × concurrency) cell carries a verdict on the time metric (wall-clock for embed, TTFT for chat_stream), (b) every cell carries the measured client→server RTT so the reader can confirm the wire was real, (c) the same shared-baseline + frozen-tuned-channel + REST-cohort topology is used for every cell (no per-cell baseline drift), and (d) the report explicitly names where tuned-gRPC wins, where REST wins, and where neither wins (`no_winner`) with supporting 95% CI-bounded numbers.

**Acceptance Scenarios**:

1. **Given** the M5.1 harness configured to run against a Modal-hosted MockEngine deployment exposing both a gRPC servicer (M3 servicers, same as M5) and a REST shim (FastAPI translating OpenAI-style request shapes into the same `MockEngine` call path), **When** the M5.1 sweep runs at hidden_size 2048, 4096, and 8192 on both paths (chat_stream, embed) at concurrencies 1, 4, and 8, **Then** the M5.1 report names a verdict per (path × hidden_size × concurrency) cell — either `tuned_grpc_recommend` (gRPC's 95% CI strictly clears REST's 95% CI in the direction of improvement), `rest_recommend` (REST's CI strictly clears gRPC's), or `no_winner` (CIs overlap) — and shows the supporting CI-bounded delta.
2. **Given** the cross-host topology produces a measured median client→server RTT in the 30–100 ms range (per M5's clarified band), **When** the M5.1 harness opens each cohort's measurement window, **Then** the measured RTT is recorded for that cohort and surfaced in the M5.1 JSON, and the harness refuses to issue verdicts on any cohort whose median RTT falls below M5's same-host-fallback threshold (i.e., the wire silently went same-region fast-path).
3. **Given** the same Modal deployment is used for the entire run (both protocols, all cells, single deploy), **When** the run completes, **Then** the deployment is torn down cleanly and the run records a single Modal app handle in the report's executive metadata so the reader can confirm the engine variable was held constant across the comparison.
4. **Given** M5 published five winning channel configurations at `embed/h2048` with deltas of −23% to −25%, **When** the M5.1 gRPC cohort uses M5's frozen-tuned channel configuration (the per-(path × hidden_size) combination of M5's per-axis winners, parallel to M5's frozen-channel baseline for schema candidates), **Then** the M5.1 report records both (a) the cross-host tuned-gRPC vs REST delta and (b) the cross-host tuned-gRPC vs default-gRPC delta (carried forward from M5's published numbers) so a reader can attribute the gRPC-vs-REST gap to encoding-and-framing vs channel-tuning components independently.

### Edge Cases

- **REST shim methodological skew.** The M5.1 REST cohort is driven through a FastAPI shim against the same `MockEngine`. The shim's HTTP/1.1 framing, JSON serialization path, and uvicorn worker model can introduce overhead that is shim-specific rather than REST-protocol-specific. The M5.1 harness MUST measure the REST shim's intra-process overhead (FastAPI handler entry → `MockEngine.generate` / `MockEngine.embed` call boundary) and record it per-cohort so the reader can confirm REST's measured time is dominated by transport, not by shim plumbing.
- **MockEngine does not produce variable inference cost.** Real vLLM's inference cost scales with prompt length, output tokens, and model size; MockEngine produces deterministic, near-zero inference cost. This is *intentional* (the same isolation-of-protocol-effects discipline M3/M4/M5 use) but creates an analytical pitfall: at concurrency levels where real vLLM would be GPU-saturated, MockEngine is not, so M5.1's c=8 numbers should be read as "wire+framing+concurrency behavior with the engine held neutral" rather than "production c=8 numbers." The M5.1 report MUST surface this read-instruction prominently in its executive section.
- **REST and gRPC both fall into `server_bound`.** Large-payload cells (typically `embed/h8192`) saw M5's `server_bound` classifier fire because server-side serialization dominated wall-clock at ~512 KB payloads. The M5.1 harness MUST apply the same `server_bound` classifier to *both* protocols' cohorts independently and MUST refuse to issue a head-to-head verdict on a (path × hidden_size × concurrency) cell where either cohort is `server_bound` — the comparison is not meaningful if server cost dominates either side.
- **Real-wire RTT is too low to matter.** If the chosen Modal region produces a measured median RTT below M5's documented exercise-threshold (default 20 ms), the head-to-head loses its "real-wire" character. The M5.1 harness MUST publish the measured RTT and attach M5's `low_rtt_caveat` annotation to any cell whose median RTT falls below the exercise-threshold; the executive narrative MUST NOT cite a low-RTT cell as the headline cross-host evidence.
- **REST cohort fails on the same Modal app while gRPC succeeds (or vice versa).** A partial-protocol failure mid-sweep — e.g., the FastAPI shim 502s while the gRPC servicer responds normally — MUST cause the affected cell to be reported as `comparison_unavailable` with the failure reason, not silently emit a one-sided cohort that would distort the head-to-head delta.
- **Concurrency interacts with HTTP/1.1 vs HTTP/2 multiplexing.** REST is held to HTTP/1.1 to match M1's methodology; gRPC uses HTTP/2 multiplexing. At c=8, the REST cohort either opens 8 connections or queues, while gRPC multiplexes on a single connection. The M5.1 report MUST surface, per cohort, the number of REST connections opened and the gRPC connection count so the reader can reason about the framing-level component of any observed delta separately from the encoding-level component.
- **README update lands before M5.1 numbers are actually in.** A merge-order accident could land the README update commit before the report files exist on the branch (e.g., a rebase reorders commits). The M5.1 PR MUST be configured so that the README update commit is the **last** commit on the branch at the moment the PR is opened; if a rebase reorders this, the harness's pre-PR checklist (US2) MUST refuse to mark M5.1 "delivered" in the narrative until the report files are confirmed present on `HEAD`.

## Requirements *(mandatory)*

### Functional Requirements

#### Cross-host harness reuse (parallels M5)

- **FR-001**: The M5.1 harness MUST reuse M5's cross-host machinery without methodological change — the Modal deploy-once endpoint provider, the per-cohort RTT probe, the `server_bound` classifier, the no-pacing MockEngine, the shared-baseline orchestrator, the borderline-expand cascade, the per-cohort CV recording, and the TTFT-first-class verdict logic. M5.1 introduces a second protocol cohort (REST) against the same Modal endpoint; it does NOT introduce new measurement methodology.
- **FR-002**: The Modal app variant used by M5.1 MUST expose **both** the M3 gRPC servicers (the same surface M5 used) **and** a REST shim (e.g., FastAPI) whose handlers translate OpenAI-style chat-completion and embedding request shapes into the same `MockEngine.generate` / `MockEngine.embed` call paths the gRPC servicers use. Both protocols MUST therefore exercise the **identical** in-container engine code path; the only operative difference between a REST cohort and a gRPC cohort on the same (path × hidden_size × concurrency) cell is the wire protocol and the channel configuration.
- **FR-003**: The Modal app MUST run on a CPU-only instance class (per M5's clarified instance choice) — MockEngine has no GPU code path and a GPU-class instance would only add cost and scheduling jitter. The M5.1 report MUST record the Modal region and instance class used so a reader can confirm topology continuity with M5.
- **FR-004**: The M5.1 harness MUST measure and record the client→server RTT (median, p95) per cohort by an active probe against the same channel (gRPC) or the same HTTP/1.1 connection pool (REST) immediately before the cohort's measurement window opens. The harness MUST refuse to issue a head-to-head verdict on any cohort whose measured median RTT falls below the documented same-host-fallback threshold (default 1 ms; reusing M5's CLI flag) AND MUST attach a `low_rtt_caveat` annotation to any cell whose measured median RTT falls below the documented exercise-threshold (default 20 ms; reusing M5's CLI flag).
- **FR-005**: The M5.1 harness MUST apply M5's `server_bound` classifier to *each protocol's cohort independently*. A (path × hidden_size × concurrency) cell where *either* the gRPC cohort or the REST cohort is `server_bound` MUST be reported as `comparison_unavailable` with the reason and MUST be excluded from `recommend` tallies and from the "Supersedes M1 (time-axis)" table.

#### Tuned-gRPC frozen-channel baseline (M5.1 gRPC cohort configuration)

- **FR-006**: The M5.1 gRPC cohort MUST use a per-(path × hidden_size) **frozen-tuned-channel configuration** composed of M5's per-axis winners at that (path × hidden_size). The construction rule is parallel to M5's frozen-channel baseline for schema candidates (M5 FR-010): take the per-axis `recommend` verdicts from M5's published report at the matching (path × hidden_size) and compose them into a single channel configuration. Where M5 emitted `no_winner` on an axis at a given (path × hidden_size), the M5.1 gRPC cohort MUST use the M1-default value for that axis at that cell.
- **FR-007**: The M5.1 gRPC cohort MUST also be measured against a per-cohort **default-config gRPC control** so the report can attribute the gRPC-vs-REST delta to the encoding-and-framing component (default-gRPC vs REST) and the channel-tuning component (tuned-gRPC vs default-gRPC) independently. The default-config gRPC cohort MAY be measured once per (path × hidden_size × concurrency) and reused across the run (single warm-up + single measurement), since M5 has already established that tuned-gRPC beats default-gRPC on cross-host real wire.

#### REST cohort plumbing

- **FR-008**: The M5.1 REST cohort MUST be driven by the existing benchmark harness's REST client path (the same path M1 used for its REST baseline), pointed at the Modal app's FastAPI shim endpoint. REST connections MUST use HTTP/1.1 (per M5.1's held-constant methodology choice; HTTP/1.1 matches M1's REST baseline). HTTP/2-over-TLS is **not** measured for REST in M5.1.
- **FR-009**: The M5.1 REST cohort's request payloads MUST be the **JSON / base64 equivalent** of the same prompt corpus the gRPC cohort uses on the same cell — i.e., the corpus is held constant, the encoding is the operative variable. Specifically: the chat path uses the same prompt strings; the embed path uses the same input tensor bytes, with the REST cohort base64-encoding them in the JSON body (matching M1's methodology) while the gRPC cohort transmits them as proto `bytes`.
- **FR-010**: The M5.1 harness MUST measure and record, per REST cohort, (a) the FastAPI shim's intra-process overhead (handler-entry-to-engine-call timing, server-side), (b) the number of HTTP/1.1 connections opened by the client at the configured concurrency, and (c) the JSON serialization byte count per request and per response. These per-cohort fields MUST be carried in the M5.1 JSON report so a reader can attribute any REST-side time component to plumbing vs transport.

#### Sweep matrix and statistical bar

- **FR-011**: The M5.1 sweep MUST measure both paths (`chat_stream`, `embed`) × three canonical widths (hidden_size 2048, 4096, 8192) × three concurrency levels (c=1, c=4, c=8). The concurrency levels are chosen to match M1's published sweep so M5.1's verdicts are directly comparable to M1's published time-axis cells (US3's supersession table depends on this matrix overlap).
- **FR-012**: Per-cohort sample sizes MUST follow M5's cascade: default n ≥ 100 measured requests per cohort; expand to n ≥ 250 on borderline (95% CIs touch). Warm-up cohorts MUST be discarded (per M5's first-cohort skew rule).
- **FR-013**: Each `tuned_grpc_recommend` or `rest_recommend` verdict MUST be supported by the candidate cohort's 95% CI strictly clearing the other protocol's 95% CI in the direction of improvement (lower bound > upper bound for maximizing metrics; upper bound < lower bound for minimizing metrics), using the same statistical rule M5 and M4 use.

#### Reporting

- **FR-014**: The M5.1 report MUST be published at `docs/benchmarks/m5_1-rest-vs-grpc.{md,json}` as a sibling to M5's cross-host report. The JSON schema MUST be **additive to M5's schema** (new fields permitted; no renames, no removals, no semantic redefinition of existing fields) so M5-aware tooling continues to work without modification. Additive M5.1-specific fields include at minimum `rest_cohort`, `tuned_grpc_cohort`, `default_grpc_cohort`, `comparison_verdict`, `comparison_unavailable_reason`, `rest_connections_opened`, `rest_shim_overhead_ms`, and `supersedes_m1_time`.
- **FR-015**: The M5.1 report's Markdown MUST include (a) an executive section naming the headline finding plus the MockEngine read-instruction (per Edge Cases), (b) the per-(path × hidden_size × concurrency) comparison matrix with verdicts, (c) the "Supersedes M1 (time-axis)" table (US3), (d) the M1 bytes-axis claims explicitly framed as **not** superseded, and (e) a "Negative results — do not re-run speculatively" appendix listing any cells whose verdict was `no_winner` or `comparison_unavailable` with the supporting numbers (Constitution V).
- **FR-016**: A reader of the M5.1 report MUST be able to determine, without reading the harness source, (a) which protocol the report recommends per (path × hidden_size × concurrency) cell, (b) which M1 time-axis cells M5.1 supersedes and which it confirms, (c) the measured RTT range across the run, (d) the Modal region and instance class used, and (e) the read-instruction caveat about MockEngine's neutral inference cost.

#### Supersession (US3)

- **FR-020**: The M5.1 report MUST include a "Supersedes M1 (time-axis)" table that maps each M1 time-axis cell the M5.1 matrix covers to (a) the M1 verdict (paraphrased into M5.1's verdict literal where M1's framing differs), (b) the M5.1 verdict literal, (c) the M5.1 supporting 95% CI-bounded numbers, and (d) a one-line rationale. ### Key Entities *(include if feature involves data)*

- **M5.1 REST cohort**: A per-(path × hidden_size × concurrency) measurement of n ≥ 100 requests driven over HTTP/1.1 against the Modal app's FastAPI shim, with the same prompt corpus as the gRPC cohort, the same warm-up rule, the same RTT probe, and the same `server_bound` classifier applied independently.
- **M5.1 tuned-gRPC cohort**: A per-(path × hidden_size × concurrency) measurement of n ≥ 100 requests driven over the per-(path × hidden_size) frozen-tuned-channel configuration composed from M5's per-axis winners, with all other harness settings matching the REST cohort.
- **M5.1 default-gRPC control cohort**: A per-(path × hidden_size × concurrency) measurement of n ≥ 100 requests driven over a default-config gRPC channel (the M1 baseline channel configuration) against the same Modal app, used to attribute the gRPC-vs-REST delta to encoding-and-framing vs channel-tuning components independently.
- **Comparison verdict**: A per-cell literal — `tuned_grpc_recommend`, `rest_recommend`, `no_winner`, or `comparison_unavailable` — emitted by the M5.1 harness for each (path × hidden_size × concurrency) cell on the time metric. Direction-of-improvement follows the metric's convention (lower bound > upper bound for maximizing metrics; upper bound < lower bound for minimizing metrics).
- **M5.1 supersession entry**: A single row in the M5.1 "Supersedes M1 (time-axis)" table that names an M1 time-axis cell, the M1 verdict (paraphrased), the M5.1 verdict, the supporting numbers, and a one-line rationale. Parallel to M5's "Supersedes M4" entries.
- **REST shim overhead record**: A per-REST-cohort record of the FastAPI handler-entry-to-engine-call timing (server-side, intra-process), the number of HTTP/1.1 connections opened by the client at the configured concurrency, and the JSON serialization byte count per request and per response.
- **MockEngine continuity invariant**: A property of the M5.1 run captured in the report's executive metadata stating that both protocols hit the same `MockEngine` no-pacing call path — i.e., the in-container engine code is identical between cohorts, so the engine variable is held constant across the head-to-head.

## Assumptions

- **M5 closed before M5.1 begins.** M5's cross-host validation merged on 2026-05-11 (PR #21, branch `017-m5-cross-host-validation`). M5's per-axis winners and frozen-channel baselines exist and are citable. The M5 published report stays in place — M5.1 records supersession against M1's time-axis claims, not against M5.
- **Mock-engine continuity.** M5.1 reuses MockEngine no-pacing on a CPU-only Modal instance class, consistent with M3/M4/M5. Real-vLLM head-to-head is **deferred to M7**.
- **Corpus continuity.** Prompt corpus follows M5's defaults (the same fixtures M3/M4/M5 use). Corpus diversity is **deferred to M6**.
- **HTTP/1.1 for REST, held constant.** REST is measured over HTTP/1.1 to match M1's methodology. HTTP/1.1-vs-HTTP/2 for REST is **out of scope** for M5.1.
- **Concurrency levels match M1.** The M5.1 sweep matrix uses c=1, c=4, c=8 — the same concurrency band M1 published — so M5.1's per-cell verdicts are directly comparable to M1's published time-axis cells (US3's supersession table depends on this matrix overlap).
- **Frozen-tuned-channel construction.** The M5.1 gRPC cohort uses a per-(path × hidden_size) frozen-tuned-channel composed of M5's per-axis winners. Where M5 emitted `no_winner` on an axis at a given (path × hidden_size), the M5.1 gRPC cohort uses the M1-default value for that axis at that cell. Constructing the frozen-tuned-channel is mechanical (deterministic from M5's published JSON) and requires no new measurement.
- **Default-gRPC control via single warm-up + single measurement.** M5 has already established that tuned-gRPC beats default-gRPC on cross-host real wire; the M5.1 default-gRPC control cohort is therefore measured once per cell and reused, not re-measured per (axis × candidate) sub-cell. The measured default-gRPC numbers may also be cross-checked against M5's shared-baseline numbers as a sanity check.
- **Bytes-axis claims remain in force unchanged.** M1's chat 89% response-byte reduction and embed 25% request-byte reduction are structural (encoding choice) and not transport-condition-dependent. M5.1 records per-cohort bytes for sanity but does **not** re-derive the canonical bytes story — the M1 claim stands.
- **Single Modal deploy per run.** The Modal app variant exposes both protocols under a single deploy and is reused across all (path × hidden_size × concurrency) cells in a run. Per-cell deploys would multiply Modal cost ~50× and are explicitly avoided (parallel to M5's `make_frozen_endpoint_provider` decision).
- **TLS continuity with M5.** The Modal app uses Modal-managed TLS (HTTPS) for the REST shim. For gRPC, plain-TCP via `modal.forward(unencrypted=True)` is used because Modal's HTTPS edge does not negotiate h2 ALPN (the same constraint M5 ran into). TLS overhead is constant within each protocol and not varied as part of any cohort's sweep.
- **Authentication continuity.** Bearer-token-in-metadata for gRPC (per M5 FR-002); equivalent bearer-token-in-Authorization-header for REST. Tokens are sourced from `MODAL_BENCH_TOKEN` per M5 conventions and are never embedded in the report.
- **Constitution V (negative-results-published) applies.** Cells whose comparison verdict is `no_winner` or `comparison_unavailable` are recorded with full supporting numbers in a named appendix so the negative result is reusable rather than re-discovered.
- **Adoption is a separate change.** A `tuned_grpc_recommend` or `rest_recommend` verdict in M5.1's report does not automatically modify production defaults, client preference, or proxy routing. Adoption is performed in a follow-up change with its own review.
- **Pre-PR README update is enforced procedurally, not by tooling.** The "README update is the last commit before `gh pr create`" rule is a procedural requirement (FR-017 / FR-019) the maintainer follows; the harness does not block PR creation. If this conflicts with established Spec Kit hook flow (e.g., a post-implement auto-commit lands additional changes), the maintainer MUST squash or reorder so the narrative-refresh commit ends up last on the branch at PR-open time.
