## Background

M5.1 (delivered 2026-05-11, PR #22) drove REST and gRPC head-to-head against the same Modal eu-west-1 MockEngine on real wire and produced 48 verdicts across an 18-cell matrix. It surfaced two questions it could not answer at the iteration count and transport surface it ran with:

1. **Production-equivalent REST transport.** M5.1 forced both protocols through Modal's plain-TCP tunnel (`modal.forward(..., unencrypted=True)`) because a smoke run measured a ~2× RTT gap between Modal's HTTPS edge (TLS-terminated, anycast-routed near the client) and plain-TCP, which would have dominated every verdict if one protocol travelled one path and the other travelled the other. Plain-TCP was the only fair transport for the head-to-head, but production REST traffic almost always travels the HTTPS edge — so M5.1's REST cohort is not the REST cohort an operator will deploy. M5.2 measures **both** REST transports against the same gRPC baselines so the operator can see how the network path changes the verdict, and so M5.1's tally can be reread against a production-equivalent REST baseline.
2. **Tuned-vs-default at higher resolution.** M5.1's per-cell deltas between `default_grpc` and the tuned cohorts (`tuned_grpc_multiplexed`, `tuned_grpc_channels`) were frequently within ±3% — at the noise floor of an n=100 sample. The "M5's tuned channel config provides no measurable benefit over M1-default on this path" finding is one of M5.1's three headline conclusions, but n=100 is not enough to distinguish "genuinely neutral" from "real but smaller than the noise." M5.2 raises n to 250 per cohort so the small deltas can be resolved against noise.

M5.2 is **methodology continuity, not new methodology**. It reuses M5.1's harness end-to-end (Modal eu-west-1 MockEngine, no-pacing, shared baseline, per-cohort RTT probe, `server_bound` classifier, additive JSON schema) and widens the cohort surface from four (REST + 3 gRPC) to five (REST × 2 transports + 3 gRPC), increases per-cohort n from 100 to 250, and adds explicit network-path naming on every verdict row so a reader cannot accidentally interpret a `rest_plain_tcp` vs `tuned_grpc_multiplexed` row as a production-equivalent comparison.

This specification also bundles two **documentation refactors** that have been deferred for several milestones and become disproportionately valuable as M5.2's report lands:

- **Consolidate running data analysis into a single top-level file.** The cumulative "what we have learned" narrative across phases is currently scattered across `README.md` (milestone roadmap with embedded findings), `docs/PLAN.md` (milestone roadmap with embedded findings — partially duplicating the README), and `docs/benchmarks/summary.md` (phase-by-phase numbers, but stopped accumulating after M3 — does not cover M4/M5/M5.1). A reader who wants the running analysis has no single canonical destination. M5.2 moves the running analysis to a new top-level file so each milestone PR has one obvious place to update.
- **Simplify README.md.** With the running analysis consolidated elsewhere, the README returns to its M1-era role: project goals, the wire-overhead thesis, the three access paths, and a high-level pointer to each phase. The README is also validated end-to-end against the current state of the repo so tooling references (Make targets, script paths, env vars, demo commands) do not drift away from reality.

The two refactors are scoped to land in the M5.2 PR — the same operator who refreshes the executive narrative for M5.2 delivery (per M5.1 FR-017's procedural pattern) is best placed to perform the consolidation in the same pass, and the alternative (a separate "doc cleanup" PR) would either reintroduce drift before it merged or block on M5.2's report numbers anyway.

### Edge Cases

- **Modal HTTPS edge does not negotiate h2 ALPN.** Modal's HTTPS edge terminates TLS but does not negotiate HTTP/2 ALPN for general-purpose containers, so gRPC over the HTTPS edge is not a credible cohort to measure. The M5.2 cohort surface MUST therefore omit `tuned_grpc_https_edge` / `default_grpc_https_edge` — the only credible gRPC transport in this milestone is plain-TCP via `modal.forward(unencrypted=True)`. The report's executive section MUST state this constraint explicitly so a future reader does not interpret M5.2's silence on an HTTPS-edge gRPC cohort as an oversight. If Modal exposes a TLS-terminated gRPC edge in a future region, that becomes a follow-up milestone, not a partial addition to M5.2.
- **HTTPS edge introduces a TLS handshake cost on the first RPC.** A cold HTTPS edge connection incurs a TLS handshake; subsequent requests reuse the keep-alive connection. The M5.2 harness MUST configure the REST client's HTTP/1.1 connection pool identically for `rest_https_edge` and `rest_plain_tcp` (`max_keepalive_connections=c`, `max_connections=c`, `keepalive_expiry=300s`, per M5.1 FR-008) AND MUST issue a warm-up batch before each cohort's measurement window opens (per M5.1's first-cohort skew rule) so the TLS handshake is not the first measured request. The report MUST record the warm-up batch size per cohort.
- **HTTPS-edge anycast routing varies by client geography.** Modal's HTTPS edge is anycast-routed and the client's measured RTT depends on the client's geographic proximity to the nearest Modal edge POP, which can differ materially from the RTT to the chosen Modal region's worker pod. The M5.2 harness MUST record the client's external IP geolocation (country, region) in the run metadata so a reader can reproduce or interpret the RTT delta correctly. A run from a different client geography MAY produce a different RTT delta and therefore a different verdict on the transport-only comparison family — this is expected and is what the per-cohort RTT recording exists to disclose.
- **n=250 doubles per-cohort runtime; total run exceeds budget.** M5.1's full sweep ran in ~640 s at n=100 across 4 cohorts × 18 cells. M5.2's run is 5 cohorts × 18 cells × n=250 = roughly 5/4 × 250/100 = ~3.1× the per-cell work. The M5.2 harness's runtime budget per full sweep MUST be ≤ 40 minutes of wall-clock on Modal CPU-only (target: ~25–30 min); if the realized runtime exceeds this, the operator MUST document the realized runtime in the PR description and the harness MUST emit the runtime in the report's executive metadata so a future reader can reproduce. The runtime budget is operator-triggered, not part of CI's runtime budget.
- **A cohort fails mid-sweep — `rest_https_edge` 502s while `rest_plain_tcp` succeeds (or vice versa).** A partial-cohort failure during a sweep MUST cause the affected cell to be reported as `comparison_unavailable` for verdict families that depend on the failed cohort (with the failure reason — Modal-edge 502, ECONNRESET on the tunnel, FastAPI shim exception), not silently emit a one-sided cohort that would distort the head-to-head. Other verdict families on the same cell that do not depend on the failed cohort (e.g., `default_grpc` vs `tuned_grpc_multiplexed` still resolvable when `rest_https_edge` failed) MUST continue to emit verdicts normally.
- **n=250 surfaces `default_grpc` vs `tuned_grpc_*` as a real but small delta (e.g., ±4–8%).** This is the affirmative case for "tuning is real on this path, just smaller than M5.1's resolution." The report MUST treat such cells as `tuned_grpc_*_recommend` or `default_grpc_recommend` (per direction-of-improvement) and the "Supersedes M5.1" table MUST surface them as `noise_resolved` (parallel to M5's `bound_classifier_transition`) rather than `verdict_changed` so a reader does not mistake them for verdict reversals.
- **ANALYSIS.md grows unbounded as future milestones land.** ANALYSIS.md is intended as a running document that accumulates one section per milestone. M5.2 establishes its structure (one milestone heading per major milestone, headline-finding-first prose, full numbers in the linked `docs/benchmarks/*.md` reports rather than re-stated inline) so future milestones add a section without rewriting existing ones. The file's M5.2-era length target is ≤ ~500 lines; if a future milestone pushes past that, the maintainer at that time evaluates a split (e.g., per-milestone subfolders), not M5.2.
- **`docs/benchmarks/summary.md` redirect breaks an external link the project does not own.** External documentation, blog posts, or vLLM RFC drafts may link to `docs/benchmarks/summary.md` directly. The M5.2 refactor MUST leave `summary.md` in place as a one-line redirect (not delete it), so that external links resolve to the redirect rather than 404. The redirect MUST point at `ANALYSIS.md` and MUST cite the M5.2 PR's commit SHA as the redirect's effective date so an external reader can confirm the redirect is intentional.
- **README simplification removes a section a contributor relied on.** The CI section, Environment Variables section, and Repository Structure section are pared in US3 but MUST NOT be deleted outright — each is preserved in shortened form. If a section is dropped entirely (e.g., "Benchmark Headlines" is collapsed into a paragraph), the M5.2 PR description MUST name the removal so a reviewer can flag a stakeholder dependency on the removed text.
- **Tooling reference validation surfaces a real bug.** If validating the README's tooling references reveals that, e.g., `make download-weights` no longer works or `demo/grpc-direct.py` references a removed module, M5.2 MUST fix the bug (not silently update the README to match a broken target). ## Assumptions

- **M5.1 closed before M5.2 begins.** M5.1's REST-vs-gRPC head-to-head merged on 2026-05-11 (PR #22, branch `018-m5-1-rest-vs-grpc`). M5.1's `tuned_grpc_*` frozen-channel construction (the per-(path × hidden_size) M5 winners composed with M1-defaults on no_winner axes) exists and is citable. M5.1's published report stays in place — M5.2 records supersession against M5.1's `rest_plain_tcp`-relative verdicts, not against the published M5.1 file.
- **Mock-engine continuity.** M5.2 reuses MockEngine no-pacing on a CPU-only Modal instance class, consistent with M3/M4/M5/M5.1. Real-vLLM head-to-head is **deferred to M7**.
- **Corpus continuity.** Prompt corpus follows M5.1's defaults (same fixtures M3/M4/M5/M5.1 use). Corpus diversity is **deferred to M6**.
- **HTTP/1.1 for both REST transports, held constant.** Both `rest_https_edge` and `rest_plain_tcp` are measured over HTTP/1.1, matching M5.1's REST baseline. HTTP/1.1-vs-HTTP/2 for REST is **out of scope** for M5.2 — the operative variable between the two REST cohorts is the network path, not the HTTP version.
- **gRPC over HTTPS edge is out of scope.** Modal's HTTPS edge does not negotiate h2 ALPN for general-purpose containers, so there is no `tuned_grpc_https_edge` cohort in M5.2 (per Edge Cases). If Modal exposes a TLS-terminated gRPC edge in a future region, that becomes a follow-up.
- **Concurrency levels match M5.1.** The M5.2 sweep matrix uses c=1, c=4, c=8 — the same concurrency band M5.1 published — so M5.2's per-cell verdicts are directly comparable to M5.1's published cells (the "Supersedes M5.1" table depends on this matrix overlap).
- **Frozen-tuned-channel construction is unchanged.** The M5.2 `tuned_grpc_*` cohorts use the same per-(path × hidden_size) frozen-tuned-channel composition M5.1 used. M5.2 does not re-tune; it measures M5's tuning at higher resolution against a production-equivalent REST baseline.
- **n=250 is the resolution increase that buys the verdict change.** Per M5.1's per-cell delta evidence, the small `default_grpc` vs `tuned_grpc_*` deltas need a sample size ~2.5× M5.1's to resolve against noise. n=250 is the target. If the M5.2 run still produces all-`no_winner` on the tuned-vs-default comparisons at n=250, that is a defensible negative result (and the headline finding for SC-004) rather than a methodology failure — the "Negative results" appendix carries the per-cohort CI bounds so the conclusion is reusable.
- **Single Modal deploy per run.** The Modal app variant exposes all five cohorts (both REST transports + three gRPC configurations) under a single deploy and is reused across all 18 cells in a run. Per-cell deploys would multiply Modal cost ~90× and are explicitly avoided (parallel to M5.1's deploy-once decision).
- **TLS continuity with M5.1.** Modal-managed TLS for `rest_https_edge`; plain-TCP via `modal.forward(unencrypted=True)` for the four plain-TCP cohorts. TLS overhead is constant within `rest_https_edge`'s cohort and not varied as part of any axis.
- **Authentication continuity.** Bearer-token-in-metadata for gRPC (per M5.1 FR-021); equivalent bearer-token-in-Authorization-header for REST. Tokens are sourced from `MODAL_BENCH_TOKEN` per M5/M5.1 conventions and are never embedded in the report.
- **Constitution V (negative-results-published) applies.** Cells whose verdict is `no_winner` or `comparison_unavailable` are recorded with full supporting numbers in a named appendix so the negative result is reusable rather than re-discovered.
- **Adoption is a separate change.** A `tuned_grpc_*_recommend`, `default_grpc_recommend`, `rest_https_edge_recommend`, or `rest_plain_tcp_recommend` verdict in M5.2's report does not automatically modify production defaults, client preference, proxy routing, or transport choice. Adoption is performed in a follow-up change with its own review.
