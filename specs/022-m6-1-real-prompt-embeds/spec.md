# Feature Specification: M6.1 — Real-Prompt-Embeddings Engine Path

**Feature Branch**: `022-m6-1-real-prompt-embeds`
**Created**: 2026-05-15
**Status**: Draft
**Input**: User description: "M6.1 as defined in docs/PLAN.md"

## Background *(non-normative summary)*

M6 published a "Supersedes M5.2 under real engine" verdict table covering 6 cells (`embed × {c=1, c=4, c=8}` and `chat_stream × {c=1, c=4, c=8}`) at fixed `hidden_size=4096`. Under M6, both the gRPC and REST embed cohorts ship opaque float32 bytes that the frontend hashes to a short text digest before submitting to the engine — i.e. both cohorts measure identical engine work (text-prompt unary completion through the embeddings endpoint) so the comparison is pure protocol/transport cost. This deliberate apples-to-apples symmetry preserves M5.2's comparison axis and answers "does M5.2's verdict survive under real engine cost?" without confounding the engine code path with the protocol cost.

M6.1 changes **exactly one variable** on the same 6-cell × 3-cohort matrix: the embed cohort engine code path. Both transports route real prompt embedding tensors through vLLM's `enable_prompt_embeds=True` engine path. The diff against M6's published JSON quantifies the engine-path cost differential per cell. This closes the workload-coverage gap for the workload that sends prompt embedding tensors to vLLM rather than text prompts, and gives any consumer choosing between the two paths a published cost differential to reason against.

M6.1 deliberately keeps the same model (Qwen/Qwen3-8B), the same hidden_size (h=4096, fixed by Qwen3-8B's architecture), the same Modal hardware (A10G eu-west-1), the same engine config (`fp16`, `max_model_len=2048`, `gpu_memory_utilization=0.92`), the same matrix (6 cells × 3 cohorts × n=100), the same classifier (5-category per-cell verdict), the same smoke gate, and the same per-cohort warmup / round-robin / retry / progress conventions. The single operative change is the engine code path the embed cohort drives.

## Clarifications

### Session 2026-05-15

- Q: What sequence length should the prompt_embeds tensor cover so the M6.1 − M6 differential is interpretable? → A: Fixed seq_len pinned to M6's text-digest tokenised length (≈8 tokens, exact value measured against the loaded model's tokenizer at sweep start and recorded in RunMeta). Holds workload size constant across M6 and M6.1 so the published "Engine path differential" is a clean read of the engine code-path cost rather than a confound with prompt-length variance. Per-RPC seed determinism applies to tensor *values* (fp16 entries via `torch.Generator(seed=M6_1_BASE_SEED + rpc_index)`); shape is fixed across all RPCs, cells, and cohorts.
- Q: When M6.1's classifier processes a cell whose M6 verdict was `verdict_buried_by_engine`, should it treat the cell as having no usable M6 baseline, or extract a cohort-mean delta from M6's data and apply the standard classifier? → A: Treat as no usable baseline (option A). Buried-by-engine cells in M6 → M6.1 classifies them as `no_winner_at_n100`, consistent with the treatment of `no_winner_at_n100` and `cell_incomplete` M6 cells (FR-010). Rationale: M6 declared the cell *unverifiable* against M5.2 precisely because engine cost swamps the signal; the same engine cost will swamp any cohort-mean delta M6.1 might extract from M6's data on that cell. Re-extracting a "winner delta" from buried-by-engine cells would smuggle a verdict claim into M6.1 that M6 itself refused to make. The "Engine path differential" section still publishes M6.1 − M6 numbers for that cell (US2 / FR-020) — operators inspect raw deltas there.
- Q: How should the harness detect and surface chat_stream control-cohort drift between M6 and M6.1, given chat_stream cells in M6.1 are structurally identical to M6 and intended as a same-period control on infrastructure noise? → A: Automated CI-overlap test against M6's published chat_stream per-cohort CIs per cell, surfaced as a `chat_stream_control_drift_warning` flag (option C). Any chat_stream cell with at least one cohort whose M6.1 CI does not overlap M6's published CI for that (cell, cohort) gets the flag in both markdown and JSON. Verdicts are still computed (the flag is diagnostic, not blocking). Mirrors `engine_cost_drift_warning` (FR-022) for UX consistency. Flag is the operator's signal to weight embed-cell verdicts more cautiously when reading the same report — if the chat_stream control shows infrastructure drift, the embed-cell engine-path differential may be partially confounded.
- Q: How should the harness handle the engine_version comparison between M6 baseline and M6.1 sweep, given the existing M6 baseline recorded `engine_version=unknown` because the version-reader helper landed post-sweep? → A: Surface the M6-recorded vs M6.1-pinned engine_version comparison in RunMeta and in a one-line methodology note in the published markdown report — informational, non-blocking (option B). RunMeta records both the M6 baseline's recorded `engine_version` (read from `m6-real-engine-mini-validation.json`'s `run_meta.engine_version` field; may be `unknown` for the legacy baseline) and M6.1's own pinned `engine_version` (read from `pyproject.toml`). The report's methodology notes name both values and warn if they differ; the operator decides whether the differential read is trustworthy. Non-blocking because the existing M6 baseline recorded `unknown` and blocking would make M6.1 unrunnable against it. Future re-published M6 baselines (with a recorded version) feed cleanly through the same plumbing.
- Q: After M6.1 publishes, what's the lifecycle of the two REST embed `input_kind` values — `prompt_embedding_b64` (M5.x / M6, raw float32 hashed to text) and `prompt_embedding_torch_b64` (new under M6.1, real prompt-embeds)? → A: Both coexist indefinitely as named, supported REST paths (option B). `prompt_embedding_b64` is preserved for M5.x / M6 reproducibility and as the cheaper text-symmetric path; `prompt_embedding_torch_b64` is M6.1's new path for callers actually shipping prompt embedding tensors. The REST contract documentation MUST name both `input_kind` values, describe their engine-side behavior, and point future callers at M6.1's published "Engine path differential" section as the decision aid for choosing between them. No deprecation, no migration mandate — M6.1 is a measurement milestone, not a refactor.

### Session 2026-05-15 (round 2 — terminology canonicalisation)

- Q: Which canonical form should the spec, plan, tasks, code identifiers, and published report use — "real prompt-embeds" or "real prompt-embeddings"? → A: "real prompt-embeds" (short form) is the canonical noun across spec, plan, tasks, code identifiers, and the published report. The fuller noun phrase "real prompt-embeddings engine path" is allowed **only in titles and one-sentence intros** (e.g. the feature-spec title and user-story titles). Reasons: (1) output paths (`m6_1-real-prompt-embeds.{md,json}`) and the branch (`022-m6-1-real-prompt-embeds`) already use this form — easier to align text to those load-bearing strings than to rename filenames; (2) code identifiers (`prompt_embedding_torch_b64`, `m6_winner_deltas`, `_resolve_prompt_embeds_input`) all trend toward `prompt_embeds` matching vLLM's `enable_prompt_embeds` API surface; (3) aligns the spec with upstream vLLM nomenclature.
- Q: Should the REST request body for `prompt_embedding_torch_b64` include explicit `dtype` and `shape` fields, or trust the torch.save bytes for metadata recovery? → A: Bytes-only — REST shim trusts torch.save serialized bytes for dtype/shape recovery (option A). `torch.save` already serializes dtype and shape into the binary; redundant JSON fields would create a place for client/server disagreement. The frontend's existing `decode_embeds` deserializes the binary and recovers dtype/shape end-to-end. Symmetric with the gRPC path (which also ships only bytes in the `prompt_embeds` proto field — no separate shape/dtype fields). Tensor shape/dtype mismatches surface as engine-side errors handled by the FR-017 `cell_incomplete` pathway.
- Q: How should the harness handle PyTorch version compatibility between the operator's client and the Modal-side frontend? → A: Pin client `torch` to the same version transitively required by the project's pinned `vllm` package and validate at driver-start (option B). The bench client's package (`tools/benchmark/pyproject.toml`) pins `torch` to the same version the server-side pinned `vllm` requires. The embed driver MUST validate `torch.__version__` against the expected pinned version at driver-start (first RPC of the first embed cohort) and exit with a clear actionable error message if mismatched. Reason: avoids silent `torch.save` / `torch.load` ZIP-format mismatches that would surface as undebuggable `decode_embeds` failures via the FR-017 `cell_incomplete` floor.
- Q: When per-cohort `n_successes` differ within a cell (because the FR-017 retry pathway allows up to 20% per-cohort attrition before `cell_incomplete`), how is the per-cell `engine_cost_mean` computed? → A: Simple unweighted average of the three per-cohort means (option A) — each cohort contributes equally to the cell-level value regardless of `n_successes`. Reasons: (1) preserves per-cohort identity that drives the cross-cohort drift check (FR-022's >10% threshold); (2) avoids biasing the cell-level number toward whichever cohort happened to have fewer Modal-flake failures (operationally arbitrary noise); (3) symmetric with the verdict classifier's CI-overlap test which operates on per-cohort CIs without re-weighting; (4) matches the M6 precedent (`aggregate_engine_cost_per_cell` in `m6_engine_cost.py`).
- Q: Should the smoke gate execute the FR-029 chat_stream control-drift check, or skip it as statistically not useful at n=10? → A: Skip FR-029 at smoke (option B). The smoke gate covers wiring validation only (cohort RPCs reach the engine and return); it does NOT run the FR-029 CI-overlap drift check because n=10 CIs are too wide to make the overlap test meaningful (a wide n=10 CI versus M6's tight n=100 CI would nearly always overlap, falsely reassuring the operator). The smoke summary MUST print a one-line stderr note explicitly stating that the chat_stream control-drift check is full-sweep-only and will run after the n=100 sweep completes. Keeps smoke within SC-002's 5-minute budget and preserves smoke's wiring-only design intent.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Resolve the "real prompt-embeddings engine path" caveat with a per-cell survival verdict (Priority: P1)

An operator who has already published M6 wants to know which of M6's per-cell REST-vs-gRPC verdicts survive when the embed cohort drives real prompt-embeds inference instead of the hashed-to-text-digest path. They run a single command and receive a published report whose headline section is a "Supersedes M6 under enable_prompt_embeds" verdict table — one row per cell — so they can answer the question for the workload that sends prompt embedding tensors to vLLM rather than text prompts.

**Why this priority**: This is the entire reason M6.1 exists as a separately-numbered milestone slice rather than a footnote. Without a per-cell categorical survival verdict against M6's baseline, the "embed cohort under M6 does NOT exercise real prompt-embeds inference" caveat (documented in `ANALYSIS.md` § M6 and in M6's published report) remains open and downstream methodology disclosures cannot reference a published cost differential.

**Independent Test**: Drive the M6.1 sweep against the project's Modal eu-west-1 region with the same Qwen3-8B model loaded on A10G. Confirm that the produced markdown report's executive section contains a verdict table covering all 6 cells, that each cell is classified into exactly one of the five canonical categories (`verdict_survives` / `verdict_changed` / `verdict_buried_by_engine` / `no_winner_at_n100` / `cell_incomplete`), and that the JSON companion file is a strict superset of M6's schema (consumable by an M6-aware reader without schema breakage, with one named additive section for the engine-path differential).

**Acceptance Scenarios**:

1. **Given** a configured Modal account, the M6.1 harness built, and M6's published JSON present on disk, **When** the operator runs the documented single-command M6.1 sweep against the eu-west-1 region, **Then** the run completes within the published runtime budget, writes a markdown + JSON report pair, and the markdown executive section contains a 6-row verdict table classifying each cell into one of the five canonical categories against M6's published per-cell baseline.
2. **Given** an M6-aware downstream consumer that already reads M6's JSON report, **When** the consumer reads M6.1's JSON output, **Then** the M6-aware consumer continues to work unmodified because M6.1's schema is a strict superset of M6's (one named additive section for the engine-path differential plus the same 5-category classifier output).
3. **Given** a cell whose M6 winner was non-overlapping CI in one direction, **When** M6.1 measures the same cell with the embed cohort on the real prompt-embeds engine path and observes non-overlapping CI in the opposite direction, **Then** that cell is classified `verdict_changed` (not `verdict_survives`).
4. **Given** a cell whose M6 verdict was `verdict_buried_by_engine` (engine cost dwarfed the M5.2 winner delta), **When** M6.1 re-measures the same cell on the real prompt-embeds path, **Then** the cell's classification against the M6 baseline is computed using the M6 winner delta (if any) under the same `engine_cost_mean ≥ 5 × |M6_winner_delta|` rule the M6 classifier used against M5.2.

### User Story 2 - Publish the engine-path differential (M6.1 − M6) per cell (Priority: P2)

The operator (or any consumer choosing between the text-prompt-completion path and the real prompt-embeds path) wants the per-cell delta between M6.1's measurements and M6's published baseline — not folded into the cohort comparisons — as a separate, named "Engine path differential" section. For each cell the differential covers both the classifier metric (per-cohort and the cell-level mean) and the per-cell `engine_cost_mean`, with units and 95% CI half-widths. **Why this priority**: This is M6.1's methodology gift to any consumer picking between the two engine paths. Without it, the consumer must read both M6 and M6.1 reports and compute the differential themselves with no published CI. P2 because the headline survival table (US1) is the milestone exit deliverable; the engine-path differential is a load-bearing methodology disclosure that turns the published reports into a directly-usable decision aid.

**Independent Test**: Read any cell's row in the published "Engine path differential" section and confirm the M6.1 − M6 deltas are present as named fields with units (ms) and 95% CI half-widths for both the classifier metric (per cohort) and `engine_cost_mean` (per cell).

**Acceptance Scenarios**:

1. **Given** a published M6.1 markdown report, **When** the operator scrolls to the "Engine path differential" section, **Then** the section names 6 cells × 3 cohorts of classifier-metric deltas (M6.1 − M6, ms, 95% CI half-widths) plus 6 cells × 1 engine-cost delta.
2. **Given** the M6 baseline JSON has `verdict_buried_by_engine` on a cell with no published M5.2 winner delta to compare against (cell was already `no_winner` in M5.2), **When** the operator reads the differential section for that cell, **Then** the section still publishes the M6.1 − M6 classifier-metric and engine-cost deltas even though no survival verdict against M6 can be computed (the cell's verdict column reads `no_winner_at_n100` or `cell_incomplete` per the classifier; the differential numbers are independent).

### User Story 3 - Catch real-prompt-embeds wiring failures cheaply with a smoke gate (Priority: P3)

The operator wants to run a fast, low-cost smoke check before committing to the full ~75–90 min Modal A10G sweep, so that a wiring bug in the new `torch.save`-encoded gRPC path, the new REST `input_kind`, or the operator's local `torch` availability surfaces in minutes rather than after a full sweep is wasted. The smoke gate exercises the same shape as M6's: 2 cells × 3 cohorts × n=10, covering both an embed cell and a chat_stream cell so a chat_stream regression introduced by the M6.1 wiring also surfaces during smoke.

**Why this priority**: Operator-quality-of-life. P3 because the failure modes the smoke gate catches are operationally cheap to recover from once detected — but expensive if discovered only after the full sweep has burned 80 minutes of Modal A10G time. The smoke gate is operator-triggered and not a CI gate.

**Independent Test**: Run the M6.1 smoke command from a freshly-checked-out workspace (with M6's baseline JSON present). Observe that it completes within 5 minutes, exits 0 if all 3 cohorts pass and 1 otherwise, and prints a per-cohort one-line summary to stderr.

**Acceptance Scenarios**:

1. **Given** a configured Modal account and M6's baseline JSON present, **When** the operator runs the M6.1 smoke command, **Then** the smoke gate exercises 2 cells × 3 cohorts × n=10 (60 RPCs total) and exits within 5 minutes with status `0` if all 3 cohorts pass on both cells.
2. **Given** the operator's local environment is missing `torch`, **When** the operator runs the M6.1 smoke command, **Then** the smoke gate surfaces the missing-`torch` failure at driver-start (before any RPC is sent) with a clear error message naming `torch` as the missing prerequisite, exits with status `2`, and prints a one-line per-cohort summary to stderr.
3. **Given** the M6 baseline JSON is missing or malformed, **When** the operator runs the M6.1 smoke command, **Then** the smoke gate aborts with a clear pre-check error message naming the failing precondition before any RPC is sent, exits with status `2`, and does not attempt the sweep.

### Edge Cases

- **Operator client machine lacks `torch`**: the gRPC embed driver cannot call `torch.save(tensor)`. Driver fails at driver-start (before the first measurement RPC) with a clear error naming `torch` as the missing prerequisite. Quickstart documentation names `torch` as a client-side prerequisite explicitly.
- **Operator client `torch` version mismatch with the server-side pinned version (FR-006)**: driver-start validation rejects the run with a clear error message naming both the detected and expected `torch.__version__` values before any RPC is sent. Prevents silent `torch.save` / `torch.load` ZIP-format mismatches from surfacing as undebuggable `decode_embeds` failures downstream.
- **M6 published JSON missing, malformed, or incomplete**: abort fast at sweep launch (and at smoke gate pre-check) before any RPC is sent. No fallback embedded snapshot. The M6 baseline is a hard prerequisite for the milestone's primary deliverable (matches M6's M5.2-precondition treatment per its FR-022-equivalent).
- **Tensor encoding/decoding mismatch (e.g. server-side `decode_embeds` raises on malformed bytes)**: counts as cohort RPC failure, retries up to 3 attempts (FR-017-equivalent), contributes to the <80-successes floor for `cell_incomplete`.
- **Real prompt-embeds engine path crashes or returns vLLM-side errors**: same failure handling as above — retry up to 3, then `cell_incomplete` if the cohort drops below 80 successes.
- **REST shim cannot serialize/forward the tensor**: same — counts as RPC failure, retries, contributes to `cell_incomplete` floor.
- **`engine_cost_mean` diverges >10% across the 3 cohorts within a cell**: the cell is annotated with an `engine_cost_drift_warning` flag (verdict still computed, per-cohort engine_cost values surfaced). This is more likely under M6.1 than under M6 because the REST shim's engine call path is structurally different from the gRPC path under the real prompt-embeds engine — the drift flag is the surface the operator uses to judge whether to trust the cell's verdict.
- **GPU memory pressure from real prompt-embeds path (extra activation memory)**: if the engine reports OOM on any cell, that cell's affected cohort RPCs count as failures, retry up to 3, then `cell_incomplete`. The engine config (`max_model_len=2048`, `gpu_memory_utilization=0.92`) is reused unchanged from M6; if real-embeds activations push past the existing KV-cache headroom for some (cell × cohort) the cell surfaces as `cell_incomplete` rather than silently degrading.
- **`prompt_embeds` tensor dtype/shape mismatch with the loaded model**: server-side error, counted as RPC failure, retries, contributes to `cell_incomplete`. Tensor shape and dtype are pinned by the driver (matching the loaded model's `hidden_size=4096` and the engine's expected fp16 dtype).
- **chat_stream cohort behaviour unchanged from M6**: chat_stream RPCs do NOT use the new prompt_embeds path under M6.1; they remain text-prompt streaming. M6.1 measures one variable change (the embed cohort engine path); chat_stream cells provide a same-period control showing whether incidental Modal/engine drift contaminated the embed-cell verdict.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The sweep MUST run 6 cells × 3 cohorts × n=100 measurement RPCs, identical to M6's matrix shape. Cells are `embed × {c=1, c=4, c=8}` and `chat_stream × {c=1, c=4, c=8}` at the fixed `hidden_size=4096` set by Qwen3-8B's architecture.
- **FR-002**: The gRPC embed cohorts (`default_grpc` and `tuned_grpc_multiplexed`) MUST encode the random prompt embedding tensor as `torch.save(tensor)`-pickled bytes and ship those bytes in the gRPC `prompt_embeds` proto field. The frontend's existing `CompletionsServicer._resolve_prompt_embeds_input` first tries `decode_embeds` (which expects the `torch.save` ZIP magic prefix), returns a real `torch.Tensor`, and the engine consumes the tensor via `enable_prompt_embeds=True`. No frontend change required.
- **FR-003**: The REST embed cohort (`rest_https_edge`) MUST send tensors encoded as base64-encoded `torch.save`-pickled bytes via a new request `input_kind` (suggested name: `prompt_embedding_torch_b64`). The REST shim MUST recognise this `input_kind`, base64-decode the payload, call `decode_embeds` to obtain a `torch.Tensor`, and pass `{"prompt_embeds": tensor}` to `engine.generate(...)` directly — driving the same `enable_prompt_embeds=True` engine path as the gRPC cohorts. The request body MUST carry the b64-encoded bytes as the sole payload field; the REST shim MUST NOT require (or honor) separate `dtype` or `shape` fields — dtype and shape are recovered from the `torch.save` serialized bytes by `decode_embeds`, symmetric with the gRPC path (which also ships only bytes in the `prompt_embeds` proto field). Tensor-shape or dtype mismatches surface as engine-side errors handled by FR-017's `cell_incomplete` retry pathway.
- **FR-004**: The existing M5.x / M6 REST `input_kind="prompt_embedding_b64"` (raw float32 bytes, hashed to text digest server-side) MUST remain functional unchanged so M5.x and M6 sweeps remain bit-reproducible. The new `prompt_embedding_torch_b64` (FR-003) and the existing `prompt_embedding_b64` MUST coexist indefinitely as named, supported REST paths — no deprecation, no migration mandate. The REST contract documentation MUST describe both `input_kind` values, name their engine-side behavior (text-digest hash vs real `enable_prompt_embeds=True`), and point readers at M6.1's published "Engine path differential" section as the decision aid for choosing between them.
- **FR-005**: chat_stream cohort RPC contracts MUST be unchanged from M6 — text-prompt streaming, no prompt_embeds path. M6.1 changes exactly one variable (embed engine path).
- **FR-006**: The operator's client machine MUST have `torch` available as a Python import for the gRPC embed driver's `torch.save()` call. The bench client's package (`tools/benchmark/pyproject.toml`) MUST pin `torch` to the same version transitively required by the project's pinned `vllm` package so client-side `torch.save` bytes are wire-compatible with the server-side `torch.load`/`decode_embeds` path. The embed driver MUST validate `torch.__version__` against the expected pinned version at driver-start (before the first measurement RPC of the first embed cohort) and exit with a clear actionable error message if mismatched, naming both the detected and expected versions. Quickstart documentation MUST list `torch` as a client-side prerequisite and name the pinned version.
- **FR-007**: All other M6 wiring MUST be reused unchanged: the same Modal app definition, the same `AsyncLLM(Qwen/Qwen3-8B, dtype=fp16, enable_prompt_embeds=True, max_model_len=2048, gpu_memory_utilization=0.92)` config, the same classifier algorithm, the same reporter, the same smoke gate framing.
- **FR-008**: The harness MUST load `docs/benchmarks/m6-real-engine-mini-validation.json` at runtime as the single source of truth for the per-cell M6 baseline (the M6 winner delta used by the classifier's 5× engine-cost rule and by the survival verdict logic), AND MUST snapshot the per-cell M6 baseline values it actually used into M6.1's JSON RunMeta under a new `m6_winner_deltas` field. Equivalent treatment to M6's `m5_2_winner_deltas` field.
- **FR-009**: The harness MUST load and validate the M6 baseline JSON before launching either smoke or full sweep — if the file is missing, isn't valid JSON, or doesn't contain entries for all 6 M6.1 cells, the harness MUST exit with a non-zero status and a clear message naming the failing precondition. No fallback embedded snapshot.
- **FR-010**: The per-cell verdict classifier MUST use the same 5-category output as M6 (`verdict_survives` / `verdict_changed` / `verdict_buried_by_engine` / `no_winner_at_n100` / `cell_incomplete`) and the same algorithm — CI-overlap test on the classifier metric, plus `engine_cost_mean ≥ 5 × |M6_winner_delta|` for `verdict_buried_by_engine` — substituting M6's per-cell winner deltas for M5.2's. Cells where M6 itself had no usable published winner delta (M6 verdict was `no_winner_at_n100`, `cell_incomplete`, **or `verdict_buried_by_engine`**) MUST classify as `no_winner_at_n100` regardless of M6.1 CI overlap. Rationale for including `verdict_buried_by_engine`: if M6's engine cost already swamped the M5.2 winner signal on that cell, the same engine cost on the same hardware will swamp any cohort-mean delta extractable from M6's data — re-extracting a "winner delta" would smuggle a verdict claim that M6 itself refused to make. The Engine Path Differential section (FR-020) still publishes the cell's raw M6.1 − M6 deltas for inspection.
- **FR-011**: The classifier metric for chat_stream cells MUST be client-observed TTFT (matching M6); the classifier metric for embed cells MUST be total per-RPC wall-clock (matching M6).
- **FR-012**: The smoke gate MUST exercise 2 cells × 3 cohorts × n=10 (60 RPCs total), covering `embed × c=1` AND `chat_stream × c=1` — identical to M6's smoke shape. The smoke gate MUST exit `0` if all 3 cohorts pass on both cells; `1` if any cohort fails an RPC; `2` if either pre-check fails — the M6 baseline JSON pre-check (FR-013) OR the client `torch` pre-check (FR-006). (Smoke collapses both pre-check failures under code `2`; full-sweep mode splits them into codes `1` and `2` respectively per FR-025 / `contracts/cli.md`.) The smoke gate MUST NOT execute the FR-029 chat_stream control-drift check (n=10 CIs are too wide for the CI-overlap test to produce meaningful signal); instead, the smoke summary MUST print a one-line stderr note stating that the drift check is full-sweep-only and runs after the n=100 measurement sweep.
- **FR-013**: The harness MUST validate the M6 baseline JSON in the smoke gate as a pre-check (in addition to the full-sweep pre-check) — exit code `2` if the pre-check fails.
- **FR-014**: One engine instance MUST be loaded once at sweep start and serve all 6 cells. No reload between cells, paths, or cohorts. `cold_start_s` is a single scalar per sweep, recorded in RunMeta.
- **FR-015**: Per-cohort warmup MUST run 10 throwaway RPCs at the start of each (cell × cohort) before the 100 measurement RPCs. Warmup RPCs are discarded from all reported metrics.
- **FR-016**: Within each cell, cohorts MUST be sequenced round-robin per c-batch — `c` concurrent RPCs of cohort A, then `c` of cohort B, then `c` of cohort C, repeated `100/c` times.
- **FR-017**: Per-RPC failures within a cohort MUST be retried up to 3 attempts. CIs report at actual `n_successes`. A cell MUST be classified `cell_incomplete` if any cohort has fewer than 80 successes after retries.
- **FR-018**: Per-RPC engine cost MUST be emitted by the frontend (`engine_forward_ms` for embed, `engine_ttft_ms` and `engine_tpot_ms` for chat_stream) — via gRPC trailing metadata for the gRPC cohorts and via REST JSON top-level fields for the REST cohort. Identical instrumentation contract to M6.
- **FR-019**: Sampling seeds MUST be per-RPC deterministic: `SamplingParams.seed = M6_1_BASE_SEED + rpc_index`, with `M6_1_BASE_SEED` default `42` recorded in RunMeta, and `rpc_index` the global measurement RPC counter across the whole sweep (warmup RPCs excluded from the index sequence).
- **FR-020**: The published markdown report MUST include the same sections as M6 — executive summary, "Supersedes M6 under enable_prompt_embeds" verdict table, engine-cost-per-RPC table, per-cohort detail, methodology notes, operator reproducibility — PLUS an additional named "Engine path differential" section (US2) that publishes per-cell M6.1 − M6 deltas for the classifier metric (per cohort) and for `engine_cost_mean` (per cell), each with 95% CI half-widths.
- **FR-021**: The published JSON output schema MUST be a strict superset of M6's schema. An M6-aware reader MUST be able to consume M6.1's JSON unmodified except to access the new additive `engine_path_differential` field (and `m6_winner_deltas` in `run_meta`).
- **FR-022**: Per-cell `engine_cost_mean` MUST be the **simple unweighted average of the three per-cohort means** — each cohort contributes equally to the cell-level value regardless of its `n_successes` (i.e. `cell_engine_cost_mean = (rest_mean + default_mean + tuned_mean) / 3`, matching M6's `aggregate_engine_cost_per_cell` convention). If any two cohorts' per-cell engine_cost means disagree by more than 10%, the cell MUST be annotated with an `engine_cost_drift_warning` flag in both markdown and JSON (verdict still computed; per-cohort engine_cost values surfaced).
- **FR-023**: Progress reporting MUST emit one stderr line per (cell × cohort) pair (18 lines over the full sweep) with format `[i/18] <cell> / <cohort> — <successes>/<n> succ — <wall_ms> — ETA <minutes>m`. Stdout reserved for the final report path. Mirrors M6's progress convention.

- **FR-025**: The harness MUST be invoked via `python -m vllm_grpc_bench --m6_1 --m6_1-modal-region=eu-west-1` for the full sweep and `python -m vllm_grpc_bench --m6_1-smoke --m6_1-modal-region=eu-west-1` for the smoke gate. CLI flag set mirrors M6's `--m6` / `--m6-smoke` shape (region, base seed, model, RTT thresholds, M6 baseline path, output paths) surface as `--m6_1-*` flags.
- **FR-026**: Output paths MUST default to `docs/benchmarks/m6_1-real-prompt-embeds.md` and `docs/benchmarks/m6_1-real-prompt-embeds.json` and MUST be overridable via CLI flags (`--m6_1-report-out`, `--m6_1-report-json-out`).
- **FR-027**: The published RunMeta MUST record `engine_version` (M6.1's own pinned `vllm` version read from `pyproject.toml`; see FR-030 for the parallel M6-baseline value), `model`, `hidden_size`, `max_model_len`, `gpu_memory_utilization`, `seq_len` (the fixed prompt_embeds sequence length pinned at sweep start per FR-028), `M6_1_BASE_SEED`, `git_sha`, Modal region, run start/end timestamps, `cold_start_s`, and the snapshotted `m6_winner_deltas` — equivalent treatment to M6's RunMeta.
- **FR-028**: The driver MUST emit random prompt embedding tensors with shape `[seq_len, hidden_size=4096]` and dtype `fp16` (matching the loaded model's hidden_size and the engine's loaded weights dtype) and ship them via `torch.save(tensor)` bytes (gRPC) or base64-encoded `torch.save(tensor)` bytes (REST). `seq_len` MUST be a **fixed value across all RPCs, cells, and cohorts**, pinned at sweep start to match M6's per-RPC text-digest tokenised length under the loaded model's tokenizer (i.e. tokenise the same M6 text-digest format `embed_<hex>` against the Qwen3-8B tokenizer and use the resulting token count). The pinned `seq_len` value MUST be recorded in RunMeta. - **FR-029**: The harness MUST run a chat_stream control-drift check after **full-sweep completion** (and ONLY at full sweep — not at smoke; see FR-012). The flag is diagnostic (not blocking) — verdicts are still computed and published. - **FR-030**: RunMeta MUST record both (a) the M6 baseline's recorded `engine_version` (read from `docs/benchmarks/m6-real-engine-mini-validation.json`'s `run_meta.engine_version` field; surfaces as `unknown` for the existing legacy M6 baseline) and (b) M6.1's own pinned `engine_version` (read from `pyproject.toml`). The published markdown report MUST include a one-line note in the methodology section naming both values; if they differ (or if either is `unknown`), the note MUST flag the comparison as informational and remind the operator that the "Engine path differential" read is cleanest when both versions match. The harness MUST NOT block or abort on engine_version mismatch — non-blocking by design so M6.1 can run against the existing legacy M6 baseline.

### Key Entities *(include if feature involves data)*

- **M6.1 Cell**: A `(path, concurrency)` pair from `{embed, chat_stream} × {1, 4, 8}`. Six total. Each cell measures 3 cohorts × n=100 RPCs plus 10 warmup per cohort.
- **M6.1 Cohort**: One of `rest_https_edge`, `default_grpc`, `tuned_grpc_multiplexed`. Cohort names match M6 verbatim so per-cell verdict tables across M5.2 / M6 / M6.1 align row-for-row.
- **M6 Winner Delta**: Per-cell winning-cohort metric delta published by M6 (analogous to the M5.2 winner delta consumed by M6's classifier). Loaded at runtime from `docs/benchmarks/m6-real-engine-mini-validation.json`; snapshotted into M6.1's `RunMeta.m6_winner_deltas`.
- **Engine Path Differential**: Per-cell M6.1 − M6 numeric deltas for the classifier metric (per cohort, ms, with 95% CI half-widths) and for `engine_cost_mean` (per cell, ms, with 95% CI half-width). Published in a named section of the markdown report and as a strict-superset field in the JSON companion (US2).
- **M6.1 Verdict Classification**: One of `verdict_survives` / `verdict_changed` / `verdict_buried_by_engine` / `no_winner_at_n100` / `cell_incomplete`. Same 5-category set as M6, computed against the M6 baseline (not M5.2) per the same classifier algorithm.
- **Prompt-Embeds Driver Payload**: A `torch.Tensor` with shape `[seq_len, hidden_size=4096]` and dtype `fp16`. `seq_len` is a fixed scalar pinned at sweep start by tokenising M6's `embed_<hex>` text-digest format against the loaded model's tokenizer (recorded in RunMeta). Tensor *values* are drawn deterministically per-RPC from `torch.Generator(seed=M6_1_BASE_SEED + rpc_index)`. Shipped via `torch.save(tensor)` bytes (gRPC) or base64-encoded `torch.save(tensor)` bytes (REST). The frontend's `decode_embeds` deserialises both forms to the same in-memory `torch.Tensor` consumed by `enable_prompt_embeds=True`.

## Assumptions

- The frontend's existing `CompletionsServicer._resolve_prompt_embeds_input` correctly distinguishes `torch.save`-encoded bytes (ZIP magic `PK\x03\x04` prefix) from raw float32 bytes, returns a real `torch.Tensor` via `decode_embeds` for the former, and falls back to the text-digest hash for the latter. This was verified during M6 (M5.x reproducibility behaviour preserved); M6.1's gRPC embed cohort relies on this dispatch to route into the `enable_prompt_embeds=True` engine path.
- M6 has published its `docs/benchmarks/m6-real-engine-mini-validation.{md,json}` artifacts before M6.1 sweeps run. M6.1's harness requires the JSON companion as a hard precondition (FR-009).
- The operator's client machine has `torch` installed as a Python import. This is the same machine where `python -m vllm_grpc_bench --m6_1` runs.
- The Modal app's existing M6 engine config (`fp16`, `max_model_len=2048`, `gpu_memory_utilization=0.92`) accommodates the additional activation memory required by the real prompt-embeds engine path on Qwen3-8B at h=4096 within A10G's 24 GB VRAM. If a sweep surfaces GPU OOM on a (cell × cohort), that cell classifies as `cell_incomplete` rather than the harness aborting (FR-017).
- M6's JSON schema and verdict-classifier algorithm are stable across M6.1; M6.1 is a strict superset add, not a rewrite of M6's output shape.
- Tensor shape conventions: prompt embeddings have `last_dim == hidden_size == 4096` matching the loaded model; the sequence-length leading dimension is drawn deterministically per-RPC. Both transports ship the same tensor encoding (`torch.save` bytes, optionally base64-wrapped for REST) so engine-side behaviour is identical across cohorts.
- The single operative change (embed cohort engine code path) is the only confound between M6 and M6.1. The chat_stream cells in M6.1 serve as a control: their drift relative to M6's published CIs is detected automatically by FR-029 and surfaced as a `chat_stream_control_drift_warning` flag in the report so the operator can weight the embed-cell verdicts and engine-path differential accordingly.
- M5.x and M6 reproductions remain functional: the existing `input_kind="prompt_embedding_b64"` REST path is preserved (FR-004), and the frontend's text-digest fallback branch in `_resolve_prompt_embeds_input` is unchanged.
- Out of scope (deferred to later milestones): prompt-length variation (M7), additional models (M8), additional canonical hidden_sizes (M8), real-engine re-validation of M3/M4 channel-tuning under the embeddings path (deferred until M6.1 produces a verdict — if M6.1 returns `verdict_buried_by_engine` everywhere, channel-tuning re-validation is moot).
