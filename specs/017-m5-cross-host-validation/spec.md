# Feature Specification: M5 — Cross-Host Time-Axis Validation

**Feature Branch**: `017-m5-cross-host-validation`
**Created**: 2026-05-10
**Status**: Draft
**Input**: User description: "M5"

## Background

M4 (delivered 2026-05-10) shipped the time-axis sweep and schema-candidate measurements on a single commodity host with the gRPC client and the gRPC server connected through `127.0.0.1`. Two of M4's four channel axes — `keepalive` and `http2_framing` — depend on real round-trip latency or real bandwidth-delay-product behavior to manifest at all, and consequently M4's verdicts on those axes carry an explicit "loopback caveat" (M4 spec FR-010 / SC-002): the report names a `no_winner` outcome but flags it as physically un-decidable on a single-host topology. The other two axes — `max_message_size` and `compression` — produced verdicts on loopback that are mechanistically defensible but still benefit from a real-network sanity check, since real-wire framing behavior can shift the supporting numbers.

M5 re-runs M4's published sweep across a real network so that the channel-level recommendations rest on transport conditions where the underlying physics can actually exercise the axes under test. M5 deliberately does **not** introduce new measurement methodology; it is the cross-host validation pass for M4's existing methodology.

## Clarifications

### Session 2026-05-10

- Q: What RTT range should M5 commit to for the cross-host topology? → A: Cross-region — Modal-hosted gRPC server in a region geographically distant from the local client; typical measured median RTT 30–100 ms. This range matches `docs/PLAN.md` v4's stated target and ensures the RTT-bounded axes (`keepalive`, `http2_framing`) can mechanically manifest, which is M5's headline-validity requirement.
- Q: Which Modal instance type should the M5 mock-engine container run on? → A: CPU-only. The mock engine has no GPU code path, so A10G would be a paid idle GPU. CPU-only removes that cost, reduces server-side scheduling jitter, and shrinks the `server_bound` cohort population.
- Q: Which exposure / auth model should the M5 Modal app use? → A: Modal-managed TLS endpoint + bearer token in gRPC metadata. TLS overhead is uniform across every cohort (so it doesn't bias any axis), and a single bearer token flows through the harness's connection-string addition. Matches the project's existing M1 / M2 Modal auth pattern.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Resolve M4's Loopback-Flagged Channel Verdicts on Real Wire (Priority: P1)

A maintainer needs M4's `keepalive` and `http2_framing` axis verdicts re-measured on a topology where the gRPC client and server are not on the same host, so that the loopback caveat M4 attached to those axes (per M4 spec FR-010) can be resolved into either a defensible `recommend` or a defensible `no_winner` whose rationale is "the axis genuinely produces no measurable effect at this RTT," not "the axis cannot manifest on this topology." The full channel sweep (all four axes × three widths × two paths) is re-run so that `max_message_size` and `compression` are also sanity-checked under real-network framing.

**Why this priority**: This is the headline value of M5. M4 shipped with two channel axes whose verdicts the report itself describes as un-decidable on the test topology; until M5 lands, those cells in the M4 report cannot be cited as definitive recommendations. The other two axes' loopback verdicts are mechanistically more defensible but still under-validated, and re-measuring all four together costs no more sweep time than re-measuring two and prevents a future "we re-measured half the axes" question. P1 because resolving the loopback caveat is the entire reason M5 exists.

**Independent Test**: A reviewer reads the channel-sweep section of `docs/benchmarks/m5-cross-host-validation.md` and confirms (a) every axis × hidden_size 2048 / 4096 / 8192 × path cell carries a verdict, (b) cells that M4 flagged with the loopback caveat carry an M5 verdict that is either `recommend` (with 95% CI strictly clearing the shared-baseline) or `no_winner` (with no loopback caveat, because the caveat is no longer physically applicable), (c) the report records the measured client→server RTT alongside every cell so a reader can confirm the wire is real, and (d) every M5 cell cites its M4 predecessor by cell coordinates and shows whether M5's verdict matches, supersedes, or is `no_winner` against M4's verdict.

**Acceptance Scenarios**:

1. **Given** the M4 published report containing channel-sweep cells with `loopback_caveat: true` on the `keepalive` and `http2_framing` axes, **When** the M5 report is published, **Then** every such cell has a corresponding M5 cell with `loopback_caveat: false` and an M5 verdict that is either `recommend` or `no_winner` supported by the same 95% CI strict-clearing rule M4 uses (M4 spec FR-008).
2. **Given** the M5 channel sweep at hidden_size 2048, 4096, and 8192 across all four axes (`max_message_size`, `keepalive`, `http2_framing`, `compression`) and both paths (chat_stream, embed), **When** a reader looks for a single recommendation per axis at each width on the time metric, **Then** the M5 report names a recommended setting (or an honest `no_winner` with supporting CI-bounded numbers) per axis × width × path on the time metric.
3. **Given** the M5 sweep runs against a gRPC server hosted on a remote host (the project's existing cloud-compute provider used for M1 / M2 baselines), **When** the harness records each cohort, **Then** every cohort entry in the M5 report includes the measured client→server RTT (median and p95) and the harness refuses to issue verdicts on any cohort whose median RTT is below a threshold that would indicate the connection unexpectedly fell back to a same-host route.
4. **Given** the M5 channel sweep produces a verdict that contradicts M4's verdict on the same cell (e.g., M4 said `no_winner` under loopback caveat, M5 says `recommend`), **When** the M5 report is finalized, **Then** that cell appears in an M5 "Supersedes M4" table with the M4 verdict, the M5 verdict, the M5 supporting numbers, and a one-line rationale (e.g., "real RTT exposed a 7.1% wall-clock reduction under keepalive=enabled at hidden_size 4096").

### User Story 2 — Re-Measure Schema Candidates Against the Cross-Host Frozen Baseline (Priority: P2)

A maintainer needs M4's three named protobuf schema candidates — packed scalars on chat completion's token-id field, `oneof` flattening on the input union, and alternative streaming chunk granularity — re-measured under cross-host transport against a per-path frozen-channel baseline derived from M5's US1 winners, so that recommendations about the wire schema continue to rest on the same evidence basis as the cross-host channel-level recommendations and do not silently inherit a baseline measured under loopback.

**Why this priority**: The schema candidates are P2 (not P1) because the channel-level cross-host verdicts are the milestone's headline contribution and deliver value standalone — schema candidates are the second pass that uses the same cross-host infrastructure. Re-measuring schema candidates on real wire is needed because (a) `oneof` flattening and chunk-granularity effects can interact with HTTP/2 framing in ways that loopback masks, and (b) the M4 schema verdicts compared against an M4 frozen-channel baseline measured under loopback — once US1 ships, the canonical baseline shifts, and schema verdicts measured against the old baseline would no longer be apples-to-apples.

**Independent Test**: A reviewer reads the schema-candidates section of `docs/benchmarks/m5-cross-host-validation.md` and confirms (a) each named candidate (packed scalars, oneof flattening, chunk granularity) has a verdict at hidden_size 4096 against the per-path frozen-channel baseline derived from M5's US1 winners, (b) each verdict reports both bytes and time deltas with 95% CIs, (c) candidates where the 4096 result is `recommend` or borderline (CI bounds touch the baseline CI) are also measured at 2048 and 8192, (d) candidates that produce no measurable signal on either metric are explicitly named as negative results in a "Negative results — do not re-run speculatively" appendix (Constitution V), and (e) each schema verdict is paired with its M4 predecessor by candidate name and the report shows whether M5's verdict matches or supersedes M4's.

**Acceptance Scenarios**:

1. **Given** the per-path frozen-channel baseline derived from M5 US1's per-axis winners and the cross-host harness from US1, **When** the schema-candidate sweep runs at hidden_size 4096, **Then** each of (packed scalars, oneof flattening, chunk granularity) produces a cohort with bytes and time measurements and a verdict for each metric independently.
2. **Given** a schema candidate whose 4096 verdict on either metric is `recommend` or whose CIs are within one CI-width of the baseline (borderline), **When** the M5 report is finalized, **Then** that candidate is also measured at hidden_size 2048 and 8192 to verify the effect generalizes across canonical widths (the same 4096-first cascade rule M4 uses).
3. **Given** a schema candidate that produces overlapping CIs on both bytes and time at the canonical width, **When** the M5 report is finalized, **Then** that candidate is recorded as a negative result with the supporting CI-bounded numbers and is named in the report's "Negative results — do not re-run speculatively" appendix.
4. **Given** a schema candidate whose M5 verdict differs from its M4 verdict (e.g., M4 said `no_winner`, M5 says `recommend`), **When** the M5 report is finalized, **Then** that candidate appears in the "Supersedes M4" table alongside the channel-level supersessions with the M5 supporting numbers and a one-line rationale that names which transport-layer mechanism plausibly produced the change.

### User Story 3 — Single-Glance Comparison of M4 and M5 Verdicts (Priority: P3)

A maintainer or downstream contributor needs to be able to look at a single artifact and answer "which M4 conclusions did cross-host measurement confirm, which did it overturn, and which were unaffected" without reading two separate reports side-by-side and reconciling them by hand. The M5 report carries an explicit "Supersedes M4" table that names every M4 cell M5 supersedes, the M4 verdict, the M5 verdict, the M5 supporting numbers, and a one-line rationale — parallel to the "Supersedes M3" table M4 published.

**Why this priority**: This is reporting hygiene rather than new measurement, and it falls out almost-for-free once US1 and US2 land. P3 because the value is "shorten the future-reader's path from M5 evidence to M4 supersession" and the underlying numbers are already in the report from US1 and US2. The supersession table is the artifact that lets a reader skip the cross-reference work, not a second source of truth.

**Independent Test**: A reviewer opens `docs/benchmarks/m5-cross-host-validation.md` and the M5 JSON file, confirms a "Supersedes M4" table is present, and verifies that for every M4 cell flagged with the loopback caveat (per M4 FR-010) there is a corresponding M5 supersession row whose M5 verdict is either a non-`no_winner` recommendation or a `no_winner` whose rationale explicitly states "real RTT does not change the verdict on this axis at this width."

**Acceptance Scenarios**:

1. **Given** every M4 cell that carries the loopback caveat (typically `keepalive` and `http2_framing` cells), **When** the M5 report is published, **Then** every such cell has a corresponding row in the M5 "Supersedes M4" table.
2. **Given** the M5 report's "Supersedes M4" table, **When** a reader scans for cells whose verdict changed between M4 and M5, **Then** changed-verdict cells are visually distinguished from confirmed-verdict cells (e.g., bold, an explicit `verdict_changed: true` flag, or a separate sub-table) so the reader can identify them in under one minute.
3. **Given** the M4 published report stays in place and is not edited by M5, **When** a future reader compares M4 and M5, **Then** every M5 supersession row cites its M4 source by cell coordinates (axis, hidden_size, path) and includes the M4 verdict literal so the reader does not need to open the M4 file to follow the supersession chain.

### Edge Cases

- **Cloud host becomes the bottleneck rather than the network.** If the remote-hosted gRPC server's container scheduling, cold-start, or co-tenant noise produces a per-RPC overhead that dominates the transport contribution, the M5 sweep MUST distinguish "transport-bound" from "server_bound" cohorts — parallel to M4's `client_bound` mechanism — and refuse to issue a transport-axis verdict on a `server_bound` cohort. The M5 report MUST surface server_bound cohort counts so a reader can judge whether cross-host evidence is trustworthy.
- **Real-wire RTT is too low to exercise RTT-bounded axes.** If the chosen client / server topology produces a measured median RTT below a documented threshold (e.g., same-region, same-zone fast path), the loopback-caveat-resolution claim weakens. The M5 report MUST publish the measured RTT and explicitly state whether it is high enough to exercise `keepalive` and `http2_framing` mechanically; if RTT is below the threshold, the axis verdict carries a "low-RTT caveat" note parallel to M4's loopback caveat.
- **Cohort variance is materially worse than M4.** Real-network jitter is expected to produce higher within-cohort CV than loopback. If the M5 shared-baseline cohort's CV exceeds M4's by more than a documented multiple, the harness MUST surface a `noisy_baseline` flag on the cohort (the same mechanism M4 spec FR-005 introduced) and the report MUST surface per-cohort CV so the reader can adjudicate verdict trust without leaving the report.
- **Cold-start or first-cohort skew.** The first cohort against a freshly-started remote server can run materially slower than steady-state cohorts because of container warm-up, JIT effects, or connection pooling. The harness MUST run a warm-up cohort whose results are discarded (not counted toward shared-baseline or any candidate cohort) before any cohort whose numbers are reported.
- **Verdict stability between M4 and M5 disagrees and the rationale is not transport-related.** If an M5 cell's verdict differs from M4's on an axis whose M4 caveat was *not* loopback-related (e.g., `compression` at hidden_size 8192 flips), the M5 report MUST flag this as "unexpected supersession" and surface the M3 → M4 → M5 verdict trio so a reader can judge whether the change is a real transport-layer effect or a measurement-noise artifact.
- **Remote host is unavailable mid-run.** Network blips or remote-host restarts MUST cause the harness to fail loudly rather than silently emit a partial cohort; partially-collected cohorts MUST be discarded and the operator instructed to re-run.

## Requirements *(mandatory)*

### Functional Requirements

#### Cross-host harness mechanics

- **FR-001**: The M5 harness MUST reuse the M4 measurement harness (`vllm_grpc_bench --m4` mode, the no-pacing mock cohort, the shared-baseline orchestrator, the borderline-expand cascade, the per-cohort CV recording, and the TTFT-first-class verdict logic) without methodological change. M5 introduces a deployment topology change and a connection-string addition; it does NOT introduce new measurement methodology.
- **FR-002**: The M5 harness MUST support a connection-string addition that points the harness's gRPC client at a remote endpoint (host:port plus TLS credentials plus a bearer token attached as gRPC call-credentials / metadata, per Clarifications 2026-05-10) instead of the in-process server M4 uses (`serve_in_process`). The bearer token MUST be sourced from an environment variable or local credentials file, never embedded in the connection string written to disk or to the M5 report. When the connection-string addition is unset, the harness MUST default to M4's in-process behavior so M4 reproductions remain bit-identical. TLS overhead is constant across every cohort and therefore does not bias any axis; the harness MUST NOT vary TLS settings as part of the channel sweep.
- **FR-003**: The remotely-hosted gRPC server MUST expose the same proto surface as M4 — M4's `MockEngine` (driven in no-pacing mode, per M4 FR-004) plus M3's `M3CompletionsServicer` and `M3ChatServicer` — so that the only operative difference between an M4 cohort and an M5 cohort on the same axis × width × path is the network path between client and server. The remote server MUST NOT silently apply channel-options that the harness's per-cell channel sweep is responsible for varying. The Modal container hosting the server MUST run on a CPU-only instance class (per Clarifications 2026-05-10) — the mock engine has no GPU code path and GPU-class instances would only add cost and scheduling jitter.
- **FR-004**: For every cohort, the harness MUST measure and record the client→server RTT (median, p95) by an active probe (e.g., a unary RPC echo against the same channel) immediately before each cohort's measurement window opens, and the recorded RTT MUST be carried alongside the cohort in the M5 JSON report. The cross-host topology MUST be configured to produce a measured median RTT in the 30–100 ms range (per Clarifications 2026-05-10). The harness MUST refuse to issue a verdict on any cohort whose measured median RTT falls below a documented same-host-fallback threshold (default 1 ms; configurable via a CLI flag) AND MUST attach a `low_rtt_caveat` annotation to any cell whose measured median RTT falls below a documented exercise-threshold (default 20 ms; configurable via a CLI flag) where RTT-bounded axes are at the bottom of their mechanical operating range.
- **FR-005**: The harness MUST detect and exclude `server_bound` cohorts — cohorts whose dominant cost component is the remote server's per-RPC overhead rather than transport+serialization — using the same mechanism M4 introduced for `client_bound` (M4 FR-004), adapted to the cross-host topology. `server_bound` cohorts MUST be counted in the report's cohort tally but MUST be excluded from `recommend` tallies; the report MUST surface the `server_bound` cohort count alongside the `client_bound` cohort count.

#### Channel-axis verdicts (US1)

- **FR-006**: The M5 channel sweep MUST measure all four axes (`max_message_size`, `keepalive`, `http2_framing`, `compression`) at all three canonical widths (hidden_size 2048, 4096, 8192) and both paths (chat_stream, embed) — the same axis × width × path matrix M4 published. M5 MUST NOT subset the matrix to only the loopback-flagged axes; the goal is parity with M4 plus real-wire validity.
- **FR-007**: For every cell that M4 flagged with the loopback caveat (per M4 FR-010), the M5 verdict for that cell MUST NOT carry the loopback caveat. The verdict literal is either `recommend` (when the candidate cohort's 95% CI strictly clears the shared-baseline's 95% CI) or `no_winner` (otherwise). The "loopback caveat" annotation MUST NOT appear on any M5 cell.
- **FR-008**: The M5 channel sweep MUST use a per-path shared-baseline cohort sized at n ≥ 100 measured at the start of the run against the remote endpoint with the M1-default channel configuration, parallel to M4's shared M1_BASELINE cohort. The M5 shared-baseline MUST NOT be reused from M4 (M4's was loopback-bound) and MUST NOT be a literal copy of M3's shared-baseline.
- **FR-009**: Each `recommend` verdict MUST be supported by a candidate cohort whose 95% CI strictly clears the shared-baseline's 95% CI under the same direction-of-improvement rule M3/M4 use (lower bound > upper bound for maximizing metrics; upper bound < lower bound for minimizing metrics). Cohort sizes follow M4's cascade: default n ≥ 100, expand to n ≥ 250 on borderline (CIs touch).

#### Schema-candidate verdicts (US2)

- **FR-010**: The schema-candidate sweep MUST be evaluated against per-path frozen-channel baselines derived from M5 US1's per-axis winners (parallel to M4 FR-011, which derived its frozen-channel baselines from M4's per-axis winners). M5 MUST NOT reuse M4's frozen-channel baseline; the comparison is against a single coherent measured configuration that reflects M5's recommendations.
- **FR-011**: At minimum, the schema sweep MUST measure the same three named candidates M4 measured: (a) packed scalars on the chat completion's token-id field, (b) `oneof` flattening on the input union, and (c) alternative streaming chunk granularity. The candidate proto shapes live where M4 placed them (`proto/vllm_grpc/v1/m4-candidates/`); M5 introduces no new candidate proto shapes.
- **FR-012**: Each schema candidate MUST be evaluated on both the bytes metric and the time metric using the same 95% CI rule as FR-009. Candidates MUST be measured at hidden_size 4096 (canonical mid-width) at minimum; candidates whose 4096 verdict on either metric is `recommend` or borderline (CIs within one CI-width of the baseline) MUST also be measured at hidden_size 2048 and 8192 (the same 4096-first cascade M4 uses).
- **FR-013**: Candidates whose 95% CIs overlap the baseline's CI on both metrics MUST be recorded as negative results with supporting numbers and named in a "Negative results — do not re-run speculatively" appendix (Constitution V).

#### Reporting and supersession (US3)

- **FR-014**: The M5 report MUST be published at `docs/benchmarks/m5-cross-host-validation.{md,json}` as a sibling to M3's bytes report and M4's time report. The JSON file's schema MUST be a **strict superset of M4's `m4-time-axis-tuning.json` schema** — only additive changes are allowed (new fields permitted; no renames, no removals, no semantic redefinition of existing fields), so any tooling that reads M4's report continues to work without modification on M5's file. Additive M5-specific fields include at minimum `client_server_rtt_ms_median`, `client_server_rtt_ms_p95`, `server_bound`, `low_rtt_caveat`, and `supersedes_m4`.
- **FR-015**: The M5 report MUST include a "Supersedes M4" table that maps each M4 cell M5 supersedes (every M4 cell flagged with the loopback caveat, plus any other M4 cell whose M5 verdict differs from M4's verdict on either bytes or time) to the M4 verdict, the M5 verdict, the M5 supporting numbers, and a one-line rationale. M3's bytes report and M4's time report MUST remain in place — supersession is recorded forward-only in M5.
- **FR-016**: A reader of the M5 report MUST be able to determine, without reading the harness source, (a) which channel-level setting per axis is recommended at hidden_size 2048 / 4096 / 8192 for each path on the time metric under cross-host transport, (b) whether each named schema candidate is recommended at the canonical width under cross-host transport, (c) which M4 cells M5 supersedes and which M5 confirms, and (d) the measured client→server RTT range across the run so the reader can sanity-check that the wire was real.
- **FR-017**: The M5 report MUST follow the M2 ground-truth workflow's citation conventions, citing the cloned vLLM and grpcio sources behind any time-affecting recommendation that differs from M4's recommendation (M4's citations remain authoritative for verdicts M5 confirms; M5 adds new citations only when the cross-host evidence shifts the verdict).

### Key Entities *(include if feature involves data)*

- **M5 cross-host shared-baseline cohort**: A per-path single-per-run measurement of the unmodified M1 channel and proto configuration measured against the remotely-hosted gRPC server, sized for stable CIs (n ≥ 100), recorded once and referenced by every M5 axis × width × path cell. Replaces M4's loopback-bound shared-baseline for purposes of M5 verdicts.
- **M5 frozen-channel baseline**: The per-path channel configuration emerging from M5 US1's per-axis winners, measured under cross-host transport; serves as the reference configuration against which schema candidates are compared in M5 US2.
- **Cross-host RTT record**: A per-cohort record of the measured client→server RTT (median, p95) gathered immediately before the cohort's measurement window opens; carried alongside every cohort in the M5 JSON report and used by FR-004 / FR-005 to gate verdict emission.
- **`server_bound` verdict modifier**: A cohort-level flag parallel to M4's `client_bound` flag, set when the dominant cost component is the remote server's per-RPC overhead rather than transport+serialization. `server_bound` cohorts are excluded from `recommend` tallies (FR-005).
- **M5 supersession entry**: A single row in the M5 report that names an M4 cell, the M4 verdict, the M5 verdict, the M5 supporting numbers, and a one-line rationale (parallel to M4's "Supersedes M3" entries).
- **`low_rtt_caveat`**: A cell-level annotation set when the measured median RTT is above the harness's connection-validity threshold (per FR-004) but below a level where RTT-bounded axes (`keepalive`, `http2_framing`) are mechanically expected to manifest. Parallel to M4's `loopback_caveat` but at a different threshold; documented in the report.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Every M4 cell flagged with the loopback caveat (per M4 FR-010) carries a corresponding M5 cell with `loopback_caveat: false` and a verdict literal of `recommend` or `no_winner`. The M5 report contains zero cells with `loopback_caveat: true`.
- **SC-002**: A reader can answer "which channel-level setting reduces chat_stream time at hidden_size 2048 / 4096 / 8192 under real-wire transport?" — at all three widths, the M5 report names a recommended setting (or an honest `no_winner` with supporting CI-bounded numbers) per axis on the time metric and shows the supporting 95% CI-bounded delta. The same applies to embed total wall-clock.
- **SC-003**: A reader can answer "do any of the named protobuf schema refinements measurably reduce wire bytes or wall-clock time at hidden_size 4096 under real-wire transport?" — the report names a winning candidate, an `no_winner` verdict with CIs, or a documented negative result for each schema candidate listed in FR-011.
- **SC-004**: A reader can answer "which M4 conclusions did cross-host measurement change?" in under one minute by reading only the M5 "Supersedes M4" table and without opening the M4 report. The supersession table visually distinguishes verdict-changed rows from verdict-confirmed rows.
- **SC-005**: The M5 report can be reproduced by a single end-to-end harness run that (a) deploys the cross-host gRPC server to the project's existing cloud-compute provider, (b) runs the harness locally with the connection-string addition pointed at the deployment, and (c) tears down the deployment when the run completes. No manual coordination beyond a single CLI invocation should be required.
- **SC-006**: Every cohort in the M5 JSON report carries the measured client→server RTT (median and p95). The report's executive summary carries the run-level RTT distribution (min, median, max across all cohorts) so a reader can confirm in under thirty seconds that the wire was real, and the run-level median RTT falls within the committed 30–100 ms range (per Clarifications 2026-05-10).
- **SC-007**: A new contributor onboarding via the M5 report can identify, in under five minutes of reading, what M5 changed about the test topology relative to M4 and which M4 conclusions are superseded.

## Assumptions

- **M4 closed before M5 begins.** M4's full sweep merged on 2026-05-10 (PR #20, branch `016-m4-time-axis-tuning`). The M4 published report stays in place and is not edited by M5 — supersession is recorded forward-only in the M5 report. M5 is the next active milestone per `docs/PLAN.md` v4.
- **Real-model validation is out of scope.** The cross-host gRPC server runs M4's `MockEngine` in no-pacing mode against the configurable mock model at canonical hidden_size 2048 / 4096 / 8192. Real-model validation lives in M7.
- **Corpus diversity is out of scope.** Prompt corpus follows M4's harness defaults (M3's `m3_long_stream.json`). Corpus expansion is M6's responsibility.
- **Cloud provider continuity.** The cross-host gRPC server is hosted on the same cloud-compute provider the project already uses for M1 / M2 baselines (Modal). M5 introduces a new Modal app for the mock-engine gRPC server but reuses the existing project provisioning, billing, and authentication conventions. Switching providers is out of scope.
- **Cross-region topology.** The Modal app is deployed in a region geographically distant from the local benchmark client so that the measured median RTT lands in the 30–100 ms range (per Clarifications 2026-05-10). Same-region or same-zone Modal deployments are explicitly out of scope for primary M5 verdicts because they produce RTT too low to mechanically exercise the RTT-bounded axes (`keepalive`, `http2_framing`) — i.e., they would re-create M4's loopback-caveat situation under a different name. The specific Modal region used is recorded in the M5 JSON report alongside the measured RTT distribution.
- **No new measurement methodology.** M5 reuses the M4 harness mechanics — no-pacing, shared-baseline, TTFT-first-class verdicts, borderline-expand cascade, per-cohort CV recording, `client_bound` exclusion. M5 adds a connection-string addition, a `server_bound` cohort flag, an RTT probe, and a `low_rtt_caveat` annotation; it does not change how verdicts are computed from cohorts.
- **Existing M4 data is reused where it remains valid.** Bytes-axis verdicts from M4 (and from M3 by transitivity) are referenced in the M5 report by citation, not re-measured. Only the time-axis cells whose interpretation depends on transport topology and the schema candidates whose comparison baseline shifts under M5 require new measurement.
- **Adoption is a separate change.** A `recommend` verdict in M5's report does not automatically modify the production `proto/`, channel-options module, or client defaults. Adoption is performed in a follow-up change with its own review, and only for candidates the maintainers accept.
- **Ground-truth citations follow the M2 workflow.** New recommendations cite cloned vLLM (`~/.graphify/repos/vllm-project/vllm/`) and grpcio (`~/.graphify/repos/grpc/grpc/`) source via `cross-repo.json` for cross-repo paths and the targeted single-repo graphs for repo-specific evidence, per the project CLAUDE.md navigation rules. Pure cross-host transport-layer behavior whose source lives in grpc's C-core (HTTP/2 transport, completion queues, low-level flow control) is cited by reading source under `~/.graphify/repos/grpc/grpc/src/core/` directly, since the graph does not index that subtree (per CLAUDE.md).
- **Constitution V (negative-results-published) applies.** Schema candidates that measure as no-effect under cross-host transport are recorded with full supporting numbers in a named appendix so the negative result is reusable rather than re-discovered. M4 cells whose M5 verdict is `no_winner` with a real RTT well above the loopback-caveat threshold are also recorded with full numbers — the negative result on cross-host is itself a contribution.
- **Cost budget is bounded.** M5's full run is expected to consume materially more cloud-compute time than M4's local run because the sweep size is the same but each RPC pays real-network latency. The expected runtime budget for the full M5 sweep is documented in plan.md (target: ≤ 8 hours of wall-clock time on a Modal CPU-only instance class per Clarifications 2026-05-10) and is operator-triggered, not part of CI's runtime budget. The CPU-only choice removes A10G's per-hour cost from the M1 baseline's provisioning, since the mock engine does not exercise the GPU.
